Kevin 最深的护城河是私有方法的积累（doc 00 Q2/Q3）。但私有方法不是单一概念——它是多种不同类型知识的总和：用户的判断框架、工作习惯、项目上下文、过去的决策记录。

## 1. 认知资本的定义

### 1.1 什么是认知资本

认知资本（Cognitive Capital）是 Kevin 为用户积累的、**无法从通用 AI 获得**的工作知识总和。

通用 AI 知道的，不是认知资本。  
只有 Kevin 见过、只有用户自己验证过的，才是认知资本。

```
用户在 Kevin 里完成一次工作循环
    ↓
Kevin 观察用户的选择：接受什么、修改什么、拒绝什么
    ↓
Kevin 识别出稳定的模式、框架、习惯、决策
    ↓
Kevin 将其结构化为认知资本的六种类型
    ↓
下一次同类工作，Kevin 的准备质量更高一个台阶
```

认知资本的积累是**复利飞轮的基础**（doc 00 Q5）。每一次工作循环既完成了任务，也让下一次同类工作的启动成本更低、质量更高。

### 1.2 认知资本 vs 普通数据

| 维度 | 普通数据（Materials） | 认知资本 |
|---|---|---|
| 来源 | 用户上传的文件、连接的数据源 | Kevin 从用户行为中观察和提炼 |
| 性质 | 描述"外部世界是什么" | 描述"用户如何思考和工作" |
| 可迁移性 | 可迁移到其他工具 | 与 Kevin 深度绑定（迁移成本高） |
| 护城河深度 | 浅（数据可被导出复制） | 深（方法论无法被复制） |

## 2. 六类认知资产总览

```
┌─────────────────────────────────────────────────────────────────┐
│                     Kevin 认知资本六类资产                         │
│                                                                 │
│  [C1] 用户画像        跨 Workspace 持久  ── 谁在用 Kevin          │
│  [C2] 项目情境        Workspace 作用域   ── 当前在做什么           │
│  [C3] 行为偏好        Workspace/User     ── 怎么表达和工作         │
│  [C4] 判断框架        Workspace/User     ── 用什么框架思考         │
│  [C5] 决策日志        Workspace 作用域   ── 过去如何判断           │
│  [C6] 外部指针        Workspace/User     ── 哪些外部系统有价值      │
└─────────────────────────────────────────────────────────────────┘
```

| 类型 | 记忆性质 | 更新方式 | 确认要求 | 消费方式 |
|---|---|---|---|---|
| C1 用户画像 | 语义 | 显式 + 推断 | 可选确认 | 常驻注入 |
| C2 项目情境 | 语义 | 自动（文件变更）+ 用户编辑 | 无需确认 | 常驻注入 |
| C3 行为偏好 | 过程 | 自动（行为信号，3次触发） | 无需确认（透明标注） | 常驻注入 |
| C4 判断框架 | 过程 | 推断（信号积累，5次触发） | **必须显式确认** | 渐进式披露 |
| C5 决策日志 | 情景 | 自动事件捕获 + 用户标注 | 无需确认 | 按需检索 |
| C6 外部指针 | 语义 | 用户显式 + 自动观察 | 可选确认 | 按需召回 |

## 3. 六类资产详细定义

### 3.1 C1：用户画像（User Profile）

**是什么**：Kevin 对"用户是谁"的理解——角色、工作场景、沟通风格、长期目标。

**内容结构**：

```yaml
user_profile:
  role: "产品经理 / 自媒体作者 / 独立投资者..."   # 职业角色
  primary_scenario: "产品/项目工作"               # 主要使用场景
  communication_style:
    verbosity: "简洁"                            # 简洁/详细/看情况
    tone: "直接"                                 # 直接/委婉/技术性
    language: "中文为主，英文术语保留"
  audience_contexts:                             # 面向不同受众时的表达差异
    - audience: "工程团队"
      preference: "结论先行，细节后补"
    - audience: "管理层"
      preference: "数据 + 决策导向，少过程"
  last_updated: "2026-05-10"
```

**积累机制**：
- 用户在初始化 Kevin 时显式填写（可选）
- Kevin 从对话和工作模式中推断，在 C3/C4 积累过程中更新
- 用户可在 My Kevin 面板随时修改

**消费方式**：常驻注入 System Prompt，会话开始时冻结快照（见 §5.2）。

**MVP 范围**：用户画像支持手动填写；自动推断为 Phase 2。

### 3.2 C2：项目情境（Project Context）

**是什么**：Kevin 对"当前 Workspace 对应的项目是什么、现在到哪了"的理解。

这是一种 **file-backed** 资产——存储在项目目录的 `.kevin/cognition.md`，用户随时可见和编辑。Kevin 维护内部镜像，并与文件保持双向同步。

**内容结构（DirectoryCognitionV1）**：

```yaml
# .kevin/cognition.md YAML frontmatter
schema: kevin-cognition/v1
workspace_id: "..."
last_updated: "2026-05-10T11:00:00+08:00"
confidence: "high | medium | low"
```

```markdown
## 这个项目是什么
[一段核心描述：目标、阶段、主要利益相关方]

## 关键约束
[时间线、资源、技术边界等]

## 当前状态
[进行到哪了，最近的重要变化]

## 待决事项
[需要用户做判断的事情]

## Kevin 的理解置信度
[对哪些部分有把握，哪些不确定]
```

**积累机制**：
- 首次扫描工作目录时生成（doc 11 第一次见面体验）
- 目录文件变更时后台自动更新
- 用户在 Chat 中纠正时同步更新
- 用户直接编辑 `.kevin/cognition.md` 时同步到内部镜像

**消费方式**：常驻注入（摘要版本），完整内容按需加载。

**详细实现**：见 `11-first-encounter-spec.md` §5（file-backed cognition 架构）。

**MVP 范围**：与 doc 11 对齐。

### 3.3 C3：行为偏好（Behavioral Preference）

**是什么**：Kevin 观察到的用户在工作中稳定的行为倾向——不是"什么"的判断框架，而是"怎么"的表达习惯和工作偏好。

**与 C4 判断框架的本质区别**：

| | C3 行为偏好 | C4 判断框架 |
|---|---|---|
| 回答的问题 | "怎么做" | "用什么标准评估" |
| 内容颗粒度 | 单一维度的倾向 | 有顺序的多步骤方法 |
| 典型示例 | "Risks 写成决策导向条目" | "分析留存问题时先看同期群再看功能归因" |
| 确认要求 | 不需要，透明标注即可 | 必须用户显式确认 |

#### 3.3.1 偏好记录的五个标准

只有同时满足以下五个标准的行为，才值得记录为偏好：

| 标准 | 含义 |
|---|---|
| **可复用** | 未来同类工作中会再次出现这个场景 |
| **稳定** | ≥ 3 次一致出现，不是一次性选择 |
| **差异** | 与 Kevin 的默认行为不同（确认默认行为不值得记录） |
| **可操作** | Kevin 能基于此偏好改变下次的输出 |
| **隐含性** | 用户不会主动说出来，但 Kevin 能从行为中观察到 |

两个**否定条件**（出现则不记录）：
- 用户明确说"这次是例外 / just this once"
- 行为是被特定上下文强迫的，非自由选择

#### 3.3.2 内容结构（Rule → Why → How to apply）

参考 Claude Code feedback 记忆类型的设计，偏好记录必须包含三个字段：

```yaml
preference_record:
  id: "pref-uuid"
  dimension: "Risks block 写法"        # 偏好适用的维度
  rule: "写成决策导向条目，而非描述性列表"  # 规则本身
  why: "（Kevin 推断）让读者快速识别风险决策点，减少理解负担"  # 原因
  why_source: "inferred | user_confirmed | user_stated"  # Why 的可信度
  how_to_apply: "每个 Risk 条目包含：风险描述 + 影响范围 + 建议应对"
  scope: "space"                        # space / user
  signal_count: 4
  created_at: "..."
  last_seen: "..."
```

**重要设计原则（来自 CCR feedback 类型）**：

> 必须同时记录**接受**和**纠正**，不只记录纠正。
> 
> 如果只记录纠正，Kevin 会变得过度保守，偏离用户已验证的好模式。接受信号（"yes exactly"、连续接受同类建议、直接使用生成内容）和纠正信号同等重要。

#### 3.3.3 Why 的两阶段共建机制

Kevin 场景中的偏好信号大多是行为信号（diff 修改、Evidence 替换），不是口头说明。Why 通常是隐含的，无法直接提取。

```
阶段 1：Kevin 的推断（自动，无需用户参与）
  → Kevin 观察到 ≥3 次同向信号
  → Kevin 推断一个"我认为这是因为 X"的假设
  → why_source = "inferred"

阶段 2：用户的确认（可选，在 Preference Review 时触发）
  → Kevin 展示："我注意到你倾向于 X，我猜是因为 Y，对吗？"
  → 用户可以：确认 / 修正 / 补充 / 不回答
  → 不回答完全合法——"inferred" 的偏好仍然有价值
  → 用户确认后：why_source = "user_confirmed"
  → 用户修正后：why 更新，why_source = "user_confirmed"
```

**Kevin 能推断 Why 的情况**（中高置信度）：
- Chat 纠正中用户说明了原因（直接提取）
- 修改方向有明确的结构含义（如"总是把 Risk 移到文档前半段" → 推断"偏好风险先行"）
- 同类信号在多个场景中以相同方向出现

**Kevin 不能推断 Why 的情况**（低置信度，保留"原因未知"）：
- diff 直接接受（只知道接受了什么，不知道为什么）
- Evidence 替换但没有说明

#### 3.3.4 积累机制

- 行为信号由 doc 10 的后置沉淀行为目录（B-S-001）采集
- 同向信号累计 ≥3 次 → 自动形成 Preference Record
- 应用时 Kevin 透明标注："（按你在这个项目里的习惯）"
- 每次应用后，用户的反应继续作为信号

**详细触发逻辑**：见 `10-ai-proactive-behaviors.md` §3.2（B-S-001）。

### 3.4 C4：判断框架（Judgment Framework）

**是什么**：用户在一类工作中稳定应用的评估方法——能用 2-4 步描述的思维程序。

这是六类资产中**价值最高、护城河最深**的一类。通用 AI 不知道这些，ChatGPT Memory 也无法捕获这些——因为判断框架不是用户说出来的，而是 Kevin 从用户的判断信号中识别和提炼的。

**与 C3 行为偏好的边界判断**：

如果一个模式可以用"**在某种情境下，先做 A，再考虑 B，用 C 标准评估**"来描述——那是判断框架（C4）。  
如果只是"**我倾向于 X 而非 Y**"——那是行为偏好（C3）。

**内容结构（JudgmentFrameworkSkill schema）**：

```typescript
interface JudgmentFrameworkSkill {
  // 身份
  skill_id: string;
  version: number;
  scope: 'space' | 'user' | 'global';

  // 描述（用用户的语言，不是技术描述）
  name: string;
  applies_to: string;        // "当你处理 X 类问题时"
  description: string;       // 2-4 句，描述核心逻辑

  // 核心内容：方法步骤
  framework_steps: Array<{
    order: number;
    step_description: string;   // "先看 X"、"再评估 Y"
    key_criteria?: string;       // 用什么标准判断
    source_signal?: string;      // 这一步的来源（可溯源）
  }>;

  // 应用条件
  trigger_context: string;
  not_applicable_when: string[];

  // 来源溯源（Skill Preview 的核心）
  source_signals: Array<{
    signal_type: string;
    observed_at: string;
    description: string;    // Kevin 观察到的具体行为
    artifact_ref?: string;
  }>;
  signal_count: number;
  confidence: 'high' | 'medium' | 'low';
}
```

**积累机制**：
- 行为信号由 B-S-002 采集和聚合
- ≥5 次信号 + Kevin 能识别出有顺序/条件的方法结构 → 生成 Skill 草案
- 草案写入 `.kevin/skill-drafts/[name].md`（file-backed，用户可直接查看）
- **必须经过 Skill Preview 用户确认**才能升级为正式 Skill
- 正式 Skill 后，应用时 Kevin 透明标注并接受反馈

**"可识别方法结构"的判断标准**：
- 有顺序性（先 X 再 Y）
- 有条件性（当 Z 情况时用 W 标准）
- 跨多类信号印证（不是单一信号类型的重复）

如果不满足，形成 C3 偏好记录，不升级为 C4。

**详细产品机制（触发、Preview UX、技术对齐）**：见 `07-skill-forge-productization.md`。

### 3.5 C5：决策日志（Decision Log）

**是什么**：Kevin 对用户在工作中做出的重要判断的情景记录——不是方法论，而是"某次，面对 X 情境，做了 Y 决策，理由是 Z"的事件日志。

这是**情景记忆**，区别于 C4 的**过程记忆**（参见 Hermes L2/L3 的设计）：

| | C4 判断框架（过程记忆） | C5 决策日志（情景记忆） |
|---|---|---|
| 问题 | "该怎么做" | "当时做了什么" |
| 内容 | 可复用的方法步骤 | 特定时间点的决策记录 |
| 访问方式 | 渐进式披露（索引 → 按需加载） | 按需检索（FTS5 关键词匹配） |
| 更新方式 | 有意识地提炼和确认 | 自动追加，不删改 |
| 消费触发 | Kevin 判断当前任务需要此框架 | "上次类似情境你选了 X" |

**内容结构**：

```typescript
interface DecisionLog {
  log_id: string;
  workspace_id: string;
  timestamp: string;
  
  // 情境
  context_summary: string;        // "在评估功能 X 的优先级时"
  artifact_ref?: string;          // 关联的 Artifact
  
  // 决策
  decision: string;               // "选择了方案 A 而非方案 B"
  decision_type: 'accept' | 'reject' | 'modify' | 'defer' | 'explicit';
  
  // 理由（可选，Kevin 推断或用户标注）
  rationale?: string;
  rationale_source: 'inferred' | 'user_stated' | 'empty';
  
  // 全文检索字段
  searchable_text: string;        // 用于 FTS5 索引的合并文本
  tags?: string[];                // 用户可选标注的标签
}
```

**什么触发记录**（由 B-S-003 管理，见 doc 10）：

| 触发事件 | 记录方式 |
|---|---|
| ActionRequest 批准 / 拒绝（带明确理由） | 自动记录 |
| 用户在 Chat 中明确说"这次我选择 X 因为 Y" | 自动记录 |
| 判断框架 Skill 被用户接受或跳过 | 自动记录 |
| 用户主动点击"记录这次决策" | 显式记录 |

**不记录**：
- 纯格式偏好（已由 C3 覆盖）
- diff 直接接受（信号质量太低，噪声大）
- 没有明显决策性质的普通操作

**消费方式**：Kevin 在前置辅助中按需检索——"上次你处理类似问题时，选择了 X，这次情况类似，要参考吗？"

**存储实现**：SQLite + FTS5（参考 Hermes hermes_state.py 的会话存储架构）。与 Materials 数据库共用 src-sidecar 的 db.ts，单独的 `decision_logs` 表。

**MVP 范围**：
- ✅ ActionRequest 决策自动记录
- ✅ 用户显式"记录这次决策"
- ⚠️ FTS5 检索 + Kevin 主动引用（依赖 db 实现，待技术评估）
- ❌ 自动推断决策理由（Phase 2）

### 3.6 C6：外部指针（Reference）

**是什么**：Kevin 知道的、用户工作中有价值的外部系统入口——飞书文档、数据看板、项目主页、重要报告的链接。

区别于 Materials（Materials 是内容本身被导入），Reference 是**指向外部系统的指针**，Kevin 知道它在哪、是什么用途，但不在本地存储全部内容。

**内容结构**：

```yaml
reference:
  ref_id: "..."
  name: "Q2 OKR 飞书文档"
  url: "https://feishu.cn/..."
  type: "doc | dashboard | project | report | other"
  description: "团队 Q2 OKR 和关键结果追踪"
  workspace_id: "..."
  scope: "space | user"
  added_by: "user_explicit | connector_observed"
  last_verified: "2026-05-10"
```

**积累机制**：
- 用户显式告知（"记住这个链接"）
- Kevin 从工作流程中观察到反复引用的外部链接后建议收录
- Connector 同步时自动注册对应数据源的 Reference

**消费方式**：在前置辅助中，当当前工作与某个 Reference 相关时，Kevin 主动提示。

**MVP 范围**：用户显式添加；自动观察为 Phase 2。

## 4. 信号来源与资产积累管道

### 4.1 信号质量分级与资产映射

| 信号类型 | 质量 | 主要流向 |
|---|---|---|
| Chat 纠正（用户说"应该是 X"） | ⭐⭐⭐ | C3 偏好（含 Why）、C4 判断框架（高价值输入）|
| diff **修改后**接受（前后值均有） | ⭐⭐⭐ | C3 偏好（核心信号）、C4 判断框架（步骤推断）|
| 手动直接编辑 Block（用户授权） | ⭐⭐⭐ | C3 偏好（原始输出）、C4 判断框架 |
| ActionRequest 拒绝 + 理由 | ⭐⭐ | C5 决策日志（主要）、C4 判断框架（辅助） |
| Evidence 替换 | ⭐⭐ | C3 偏好（证据标准维度）|
| diff 反复拒绝 | ⭐ | C3 偏好（负向偏好）|
| diff 直接接受 | ⭐ | C3 偏好（弱正向确认）|
| 用户显式说明 | ⭐⭐⭐ | C1 画像、C3 偏好（含 Why）、C5 决策日志 |

### 4.2 信号到资产的提炼逻辑

```
行为信号流入
  │
  ├─ 单次信号 → C5 决策日志（如果是 ActionRequest 决策）
  │
  ├─ 同向信号累计 ≥3 次
  │   → 检查：是单一维度倾向，还是有步骤结构？
  │       ↓ 单一维度            ↓ 有步骤结构（有顺序/条件）
  │   C3 偏好记录           C4 草案候选（继续积累到 ≥5 次）
  │   （自动生效，透明标注）
  │
  └─ 信号积累 ≥5 次 + 可识别方法逻辑
      → C4 判断框架草案（B-S-002 触发，写入 .kevin/skill-drafts/）
      → Skill Preview → 用户确认 → 正式 C4 Skill
```

### 4.3 背景提炼（参考 Hermes/CCR 的 Fork Agent 模式）

信号处理和资产提炼发生在**主任务完成后的后台**，不阻塞用户当前任务：

- 使用 Fork Agent 执行 Review（独立迭代预算，静默模式）
- Fork Agent 禁用自身 Nudge（防止递归）
- 写入磁盘后，下次会话的 System Prompt 注入生效（冻结快照保护 Prefix Cache）
- 对用户的通知：只显示一行摘要（"Kevin 注意到了一些新的偏好"），不展示过程

## 5. 存储与注入架构

### 5.1 四层存储结构

```
┌───────────────────────────────────────────────────────────────┐
│ Layer 1：常驻注入（System Prompt，冻结快照）                      │
│   C1 用户画像（精简版）                                          │
│   C2 项目情境（摘要版，全量在文件）                                │
│   C3 行为偏好（全量，有硬字符上限）                                │
│   C6 外部指针（仅当前 Workspace 相关）                            │
├───────────────────────────────────────────────────────────────┤
│ Layer 2：渐进式披露（System Prompt 只注入索引，按需加载全量）         │
│   C4 判断框架 Skill（索引：名称 + 一行描述）                        │
│   → Agent 判断需要时加载完整 framework_steps                      │
├───────────────────────────────────────────────────────────────┤
│ Layer 3：按需检索（FTS5，Agent 主动触发）                           │
│   C5 决策日志（SQLite + FTS5 全文索引）                            │
│   → 触发词："上次类似情境"、"你之前对这类问题的判断"                   │
├───────────────────────────────────────────────────────────────┤
│ Layer 4：文件系统（file-backed，用户可见可编辑）                     │
│   C2 .kevin/cognition.md                                       │
│   C4 .kevin/skill-drafts/[name].md（草案阶段）                   │
└───────────────────────────────────────────────────────────────┘
```

### 5.2 冻结快照原则（Prefix Cache 保护）

**来自 Hermes MemoryStore 的关键设计**：

System Prompt 中注入的认知资本内容（Layer 1），在会话开始时拍快照冻结，**会话期间不更新**。

- 会话内的新信号 → 即时写入磁盘，**不**更新当前会话的 System Prompt
- 下次会话开始时 → 加载最新版本，重新冻结快照

原因：System Prompt 的 Prefix Cache hash 必须在整个会话中保持稳定。任何修改都会击穿 Anthropic/OpenAI 的 Prefix Cache，导致后续每轮 API 调用成本增加约 75%。

### 5.3 字符预算约束

Layer 1 常驻注入的总预算需要严格控制，参考 Hermes 的设计（3,575 字符硬上限）：

| 类型 | 建议预算 | 超出时策略 |
|---|---|---|
| C1 用户画像 | ≤ 500 字符 | 只保留核心角色和沟通偏好 |
| C2 项目情境（摘要） | ≤ 800 字符 | 保留"是什么 + 当前状态"，截断细节 |
| C3 行为偏好 | ≤ 1,200 字符 | 优先保留高频 + 高置信度偏好，按 signal_count 排序截断 |
| C6 外部指针 | ≤ 400 字符 | 只保留最近使用的 3-5 个 |
| **合计** | **≤ 3,000 字符** | |

C4 判断框架索引（名称 + 一行描述）另计，约 10 token/条 × 条数。

### 5.4 情景记忆 vs 过程记忆分离

**来自 Hermes 学习循环的核心原则**：

| | 过程记忆（C3、C4） | 情景记忆（C5） |
|---|---|---|
| 问题 | "该怎么做" | "发生了什么" |
| 存储 | Markdown 文件（可迭代更新） | SQLite 追加日志（不改写） |
| 更新方式 | Patch 优先（局部修改，不全量重写） | 只追加，不修改历史 |
| 检索 | 渐进式披露（索引优先） | FTS5 关键词检索 |

两种记忆的访问模式和更新频率完全不同，**不能混存**。

## 6. 消费接口：认知资本如何影响 Kevin

认知资本最终通过"前置辅助的质量"体现价值——Kevin 准备的认知环境越接近用户的思维框架，洞察就越容易发生。

### 6.1 注入优先级（当前上下文 > 认知资本）

认知资本是背景层，不是前景层。优先级如下：

```
P0 当前任务上下文（用户正在做什么，当前 Artifact 内容）
P1 项目情境 C2（当前 Workspace 的理解）
P2 行为偏好 C3（怎么表达，怎么工作）
P3 判断框架 C4（面对此类问题的思维框架）
P4 决策日志 C5（按需，仅在"类似情境"触发时）
P5 用户画像 C1（最稳定，最不需要实时调整）
```

认知资本不应覆盖或压制用户的当前指令。当用户明确指定某个方式时，任何认知资本层的偏好都不应覆盖。

### 6.2 应用时的透明标注

每次认知资本影响了 Kevin 的输出，必须让用户知道：

| 类型 | 标注方式 |
|---|---|
| C3 行为偏好 | "（按你在这个项目里的习惯）" |
| C4 判断框架 | "Kevin 在准备这份内容时，应用了你的「[Skill 名]」方法框架" |
| C5 决策日志 | "上次你在类似场景里选择了 X——这次情况类似，要参考吗？" |

用户可以随时说"这次不用这个"，仅本次跳过，不修改资产本身。

## 7. My Kevin：统一资产管理视图

### 7.1 设计原则

> Kevin 可以自动积累，但不能静默积累。  
> 用户必须能在任何时候看到"Kevin 记得关于我的什么"，并能修改和删除。

这是用户信任的基础。如果用户不知道 Kevin 积累了什么，信任会断。

### 7.2 入口位置

## 8. 设计护栏

### 8.1 什么不能自动记录（即使有足够信号）

| 禁止自动记录的内容 | 原因 |
|---|---|
| 任何包含个人隐私的判断（如薪资决策） | 隐私边界 |
| 面向特定第三方的负面评价（如"我觉得 X 同事能力差"） | 风险控制 |
| 从 Materials 中可直接推导的内容（如项目名称、文件结构） | 无增量价值 |
| 一次性上下文强迫的行为（如"紧急项目只能用方案 B"） | 不稳定，不值得记 |

### 8.2 共享边界

判断框架（C4）可能包含用户的核心竞争方法，默认全部私有（`scope: 'space'` 或 `scope: 'user'`）。

提升到团队共享时，必须用户**显式确认**"这个框架可以被团队成员看到"。不支持静默推送到团队层。

### 8.3 Kevin 的"记忆验证"原则

参考 CCR 的 TRUSTING_RECALL 设计：

> 认知资本中的内容是"Kevin 的观察和推断"，不是"既成事实"。
>
> 每次 Kevin 基于认知资本给出建议时，必须有透明标注，让用户知道这是基于"之前观察到的"而非"确定知道的"。
