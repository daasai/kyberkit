## 1. 第一性原理

> ⚠️ **关于"Enterprise"措辞**：本节中"Enterprise"前缀和"企业"用语反映 MVP 的首阶段聚焦（B2B 产品工作场景），不是 Kevin 的产品定义边界。Kevin 的产品定义服务更广的高强度知识工作群体，包括自媒体、个人投资、研究、咨询等场景。详见 §4 目标用户与 `00-first-principles.md` 范围与适用性。下方"Enterprise AI Semantic Control Plane"措辞待 MVP 后重写为更通用表达。

采用 AI 产品时，真正购买的不是"更聪明的回答"，而是"能否把 AI 放进真实工作而不失控"。

Kevin 2.0 的产品本质是（MVP 用语，待重写）：

```text
Enterprise AI Semantic Control Plane
企业 AI 语义工作控制层
```

它不替代模型，不替代文档平台，也不替代自动化平台。它把模型、材料、制品、系统动作和审计关系组织到一个用户可理解、可治理的工作层中。

### 1.1 认知循环与 AI/人的分工原则

高强度知识工作的本质是**在复杂系统中持续完成认知迭代循环**：

```
连接 → 获取/处理 → 洞察 → 认知迭代/决策 → 执行 → 监控
                    ↑           ↑
                （机会/偏差/风险）  （判断/方案/举措）
```

**洞察和决策是人类主权地带。** 用户的私有方法、情境判断、责任承担是不可替代的。Kevin 的角色是：

| 阶段 | AI 的角色 |
|---|---|
| 连接 → 获取/处理 | **AI 主导**：处理复杂系统的信息规模、速度和关联复杂度 |
| 洞察的**前置辅助** | **为人服务**：准备认知环境，让洞察更容易发生 |
| 决策的**后置沉淀** | **为人服务**：捕获用户判断，沉淀私有方法，作为下次基础 |
| 执行 → 监控 | **AI 主导**：意图转化为数字动作，持续监控变化 |

这个分工是 Kevin 产品设计的底层逻辑。所有功能决策，都应回到"这是在帮用户准备洞察，还是在帮用户沉淀判断，还是在替用户执行"来判断。

### 1.2 为什么 Kevin 能做到这一点，而通用 AI Chat 不能

通用 AI Chat 的根本缺陷不是"不够聪明"，而是**没有上下文积累**。

每次对话从零开始，用户必须反复喂上下文。上次的材料、上次生成的制品、上次拒绝的建议、上次签批的动作——通用 AI 全都不记得。

Kevin 的技术原理与此不同：

```text
上下文积累（Context Accumulation）
Materials + Artifacts + Audit + Work Patterns
随时间沉淀 → 用户每次工作，Kevin 比上次更了解这个项目
```

这意味着：

- **同样的 LLM，有上下文的 Kevin > 无上下文的通用 AI**
- Kevin 的竞争优势不依赖"比 Claude 更聪明"，而依赖"比 Claude 更了解你的工作"
- 用户积累越深，Kevin 越难被替代——这是结构性护城河，不是技术追赶优势

### 1.2 上下文与记忆是五项能力的统一底层

Kevin 的五项核心能力，本质上都是上下文与记忆的具体表达：

| 能力 | 实质 |
|---|---|
| 真实上下文连接 | **建立上下文**：Materials 是 Kevin 对你工作内容的持久感知 |
| 工作对象化 | **结构化上下文**：Artifact 是 Kevin 对你工作状态的语义记忆 |
| 可控执行 | **上下文感知的行动**：Kevin 知道这次 Action 作用于哪个已知对象 |
| 可追责链路 | **决策记忆**：Audit 是 Kevin 对"谁、基于什么、做了什么决定"的长期记忆 |
| 能力复利 | **工作模式记忆**：Skill Forge 记住你怎么工作，而不只是你做了什么 |

Evidence Badge 也是记忆的展示窗口——Kevin 在说："我记得这个结论是从哪里来的。"

## 2. 核心命题

Kevin 2.0 要让企业用户完成从“问 AI”到“让 AI 参与工作”的转换。

这个转换需要五项能力同时成立：

1. **真实上下文连接**：Kevin 能接触本地 Library、企业系统、外部数据、用户材料。
2. **工作对象化**：Kevin 不是只返回消息，而是生成可持续管理的 Artifact、Action 和 Audit。
3. **可控执行**：所有外部写入、发布、执行都必须经过风险标识、预览与 Sign-off。
4. **可追责链路**：用户能知道 AI 用了什么材料、生成了什么、谁批准了什么、最终写到了哪里。
5. **能力复利**：重复工作模式可以通过 Skill Forge 被沉淀为可复用能力。

智能能力是这五项的底层支撑。没有足够智能，连接、对象化和治理会变成空壳；没有治理，智能越强风险越大。

## 3. 相对竞品的定位

Kevin 与所有竞品的核心差异可以用一句话总结：

```text
不止 Chat，始终 Always-on：Kevin 在用户私域中持续服务，并在反馈中持续进化。
```

这句话的产品契约见 `13-always-on-service-contract.md`；专业工种扩展方式见 `14-worktype-pack-architecture.md`；私域与记忆治理见 `15-private-domain-memory-governance.md`。

这不是单点功能差异，而是三项组合差异：

- **服务范式差异**：从会话式响应到 Workspace 常驻的持续服务
- **数据边界差异**：从单平台或单会话上下文到私域全链路上下文
- **能力演化差异**：从预设流程到基于真实反馈的能力进化

### 3.1 相对通用 AI Chat（ChatGPT / Claude / Gemini）

| 维度 | 通用 AI Chat | Kevin |
|---|---|---|
| 核心对象 | conversation（每次重置） | workspace / material / artifact（持久积累） |
| 上下文 | 单会话 window，跨会话失忆 | 跨会话 Materials + Artifacts + Audit |
| 回答质量 | 通用智能，不了解你的工作 | 有上下文的定向智能 |
| 记忆 | 无 | 工作对象、材料引用、审批历史、工作模式 |

Kevin 不以"回答是否更聪明"作为主要差异，而以"是否在你的私域里持续在线地把工作做成，并随着反馈变得更懂你"形成差异。

### 3.2 相对飞书 AI / Notion AI（文档平台内置 AI）

这是最需要正面应对的竞争格局。飞书 AI 原生嵌入飞书文档，用户可能会问：

> **"为什么我要在 Kevin 生成内容再投影到飞书，而不是直接在飞书里用 AI？"**

| 维度 | 飞书 AI / Notion AI | Kevin |
|---|---|---|
| 上下文范围 | 当前文档或当前 Workspace 文本 | 跨文档、跨系统、跨时间的结构化材料 |
| 数据连接 | 平台内部数据 | 数仓、本地文件、外部文档、Connector 数据 |
| 内容溯源 | 无或极弱 | Evidence Badge + EvidenceRef，每个结论可追溯 |
| 工作记忆 | 无 | Artifact 历史、Diff 决策、ActionRequest 记录 |
| 模式复利 | 无 | Skill Forge 沉淀工作模式 |

Kevin 的护城河：**私域全链路上下文 + 可追责证据链 + 跨时间工作记忆 + 反馈驱动能力进化**。飞书 AI 只能看到飞书内的内容，无法访问你的数仓、本地文件和上次工作的 Artifact 状态。

### 3.3 相对 Agent IDE（Cursor / Devin）

Agent IDE 围绕代码仓库、终端和 diff 建立闭环，强调编程生产效率。Kevin 强调的是面向非编程专业工作的 **Always-on 在场服务**：在用户私域持续接入上下文，并在反馈中持续进化工作能力。

Kevin 的对象不是 repo，而是 Semantic Workspace；核心制品不是 code diff，而是 PRD、运营复盘、策略报告、行动计划等业务 Artifact。

### 3.4 相对协作文档平台（飞书 / Confluence / Notion）

飞书、Confluence、Notion 是协作与传播平面。Kevin 不正面竞争通用编辑器，而是在用户私域提供跨系统、持续在线、可进化的工作控制层。

Kevin 的定位是控制平面：

- 在 Kevin 内完成材料引用、AI 生成、审查、签批、审计（上下文积累层）。
- 在外部平台完成组织协作、评论、传播、知识库沉淀（协作传播层）。

### 3.5 相对自动化/编排平台（Dify / Coze / n8n）

| 维度 | 自动化平台 | Kevin |
|---|---|---|
| 启动方式 | 先设计流程节点 | 先完成一次真实工作 |
| 用户门槛 | 需要理解节点、触发器、流程图 | 自然语言工作，模式自动沉淀 |
| 上下文 | 流程参数，无工作记忆 | 持久 Materials + Artifacts + Audit |
| 适用人群 | 技术运营 / 开发者 | 产品经理 / 运营分析师 / 业务负责人 |

Kevin 的方向是"先完成工作，再把稳定模式沉淀为 Skill 并持续进化"，而不是"先搭流程，再执行业务"。
## 4. 目标用户

### 4.1 产品定义服务的用户群（终极形态）

Kevin 服务的不是单一职业，而是一组在复杂信息中持续产出高责任判断的**专业工种**。这类用户共同特征——上下文密集、判断频繁、有可识别的私有方法。

专业工种不是散落的用例，而应通过 WorkType Pack 产品化扩展。每个 Pack 由 Artifact Types、Connectors、Knowledge Schema、Skill Library 和 Evaluation Tasks 组成，详见 `14-worktype-pack-architecture.md`。

锚定场景包括（不穷举）：

- 产品经理、创业者、业务负责人
- 自媒体作者、内容创作者、独立研究者
- 个人投资者、资产管理者
- 数据 / 业务分析人员
- 咨询师、顾问、研究员

### 4.2 MVP 验证服务的用户群（首阶段聚焦）

MVP 在工程深度上首先服务"产品 / 项目工作"场景：

- 产品经理、创业者、业务负责人
- 运营与数据分析人员
- 需要处理跨文档、跨数据、跨系统工作的团队成员

但产品语言、第一次见面体验、差异性标签从 Day 1 起即面向 §4.1 的全体用户群——**避免被误读为"PM 专用工具"**。

> **MVP 收敛策略**：工程深度收敛 + 产品语言广度。换场景应该是"新增连接器和 Skill 库"，不是"产品改造"。这是检验产品定义是否通用的关键标准。详见 `00-first-principles.md` 的"范围与适用性"章节。

### 4.3 不在产品定义内的用户

- 只需要轻问答、轻总结、轻写作的普通用户
- 以多人文档协作为核心诉求的团队（飞书 / Notion 已经服务得很好）
- 以低代码自动化搭建为核心诉求的技术运营用户

## 5. 产品叙事

### 5.1 核心定位（一句话）

```text
不止 Chat，始终 Always-on。
```

> 注：MVP 阶段对 B2B 用户的传播版本可保留"企业 AI 工作的语义控制层"。但产品定义本身不带"企业"前缀。

配套副标题（对外传播）：

```text
Kevin 面向专业工作场景，持续连接上下文、辅助判断、推动闭环，并在反馈中不断进化。
```

### 5.2 上下文与记忆视角的叙事

面向 §4.1 全体目标用户（领域无关）：

```text
Chatbot 负责回答，Kevin 负责把工作持续做成。
一旦 Workspace 创建，Kevin 在你的授权边界内持续接入工作上下文，
连接并整理跨系统材料，辅助你形成判断，推动决策落地与结果闭环。
```

更完整的表达：

```text
Kevin 是面向专业工种的 Always-on 数字管家。
它在用户私域中持续组织 Materials、Artifacts、Actions 与 Audit，
把连接、认知辅助、执行与监控串成可追责、可复用、可进化的工作闭环。
你工作得越多，Kevin 越懂你的项目与方法。
```

英文表达：

```text
More than chat. Always on.
Kevin is an always-on digital steward for professional work.
Within your private domain, it continuously connects context, assists judgment,
drives execution to closure, and evolves from feedback over time.
```

### 5.3 产品成长叙事（"越用越聪明"）

Kevin 的价值不是第一次使用就碾压竞品，而是随着上下文积累，Kevin 对你工作的理解不断加深：

```text
第 1 次使用：基于你提供的材料完成第一份制品，每个结论可追溯
第 5 次使用：Kevin 已经知道你的工作风格和判断偏好
第 20 次使用：Kevin 能预判你需要什么，比任何通用 AI 都更了解这个项目
```

这条成长曲线是 Kevin 的结构性护城河：用户积累的上下文无法被竞品直接复制。

> 上面"PRD"等具体产物名是 MVP 验证场景的举例。同样的成长曲线在自媒体作者的内容生产、投资者的研究决策、咨询师的客户工作中都成立——不同的是材料、制品和判断框架，相同的是认知循环和上下文积累机制。

### 5.4 偏投资人叙事（战略说服力优先）

一句话定位（融资/路演口径）：

```text
Kevin 是面向专业工种的 AI Work Operating Layer：在用户私域中提供 7x24 持续服务，
从上下文接入、认知辅助到执行闭环，并通过反馈驱动能力进化。
```

核心战略差异（建议作为对外讲解主干）：

| 差异维度 | Kevin | 常见替代方案 |
|---|---|---|
| 服务范式 | **Always-on 持续服务**（Workspace 常驻） | 会话式工具（一次一问） |
| 数据边界 | **私域全链路上下文**（跨系统连接 + 授权边界内持续更新） | 单平台内上下文或单会话上下文 |
| 能力演化 | **反馈驱动能力进化**（从真实工作中沉淀偏好与方法） | 预设流程或手工编排为主 |

护城河表述（投资人关注）：

```text
Kevin 的护城河不在单次回答质量，而在持续服务过程中沉淀的工作上下文、
决策反馈与工种方法模型。这些资产与用户业务深度耦合，迁移成本随使用时间上升。
```

30 秒电梯稿（融资会谈可直接复述）：

```text
今天的大模型已经很会回答问题，但专业工作的核心难题不是回答，而是持续把事情做成。
Kevin 把 AI 从“聊天入口”升级为“Always-on 在场服务”：一旦建好 Workspace，就在用户私域持续连接跨系统上下文，
辅助判断、推动执行并闭环结果。更关键的是，Kevin 会从真实反馈中持续进化能力，
不是手工搭流程，而是工作越多越懂这个工种、这个项目、这个人的方法。
```

## 6. 设计判断

### 6.1 核心产品风险

Kevin 2.0 的核心产品风险不是"功能不够多"，而是两个：

1. **用户仍把它理解为带文件侧栏的 AI Chat**——所有 MVP 设计必须服务于让用户看到工作状态、材料、制品、动作和审计，而不是空聊天框。

2. **上下文积累优势在冷启动时不可见**——Kevin 最强的时候是使用了一段时间之后，但用户必须在第一次就感受到上下文的价值，否则他们不会坚持到第 5 次。

在差异化标签确定为"不止 Chat，始终 Always-on"之后，新增第三个风险：

3. **Always-on 承诺被理解为越权或打扰**——Kevin 必须明确持续服务的边界、用户控制权、通知频率和隐私治理。Always-on 的产品契约见 `13-always-on-service-contract.md`，私域与记忆治理见 `15-private-domain-memory-governance.md`。

### 6.2 冷启动设计原则

第一次使用的核心目标不是"配置 Workspace"，而是**帮 Kevin 快速建立初始上下文**：

```text
帮 Kevin 了解你的工作
→ 这个项目叫什么？
→ 把你最常用的几个文件拖进来
→ 你现在最需要完成什么？
```

3 步，Kevin 有了初始上下文，第一次工作就能产生可感知的价值。

### 6.3 "Kevin 记得你"的产品感知时刻

上下文积累必须有对用户可见的展示时刻，否则用户感知不到记忆在增长：

- **Evidence Badge**：Kevin 展示"我记得这个结论从哪里来"
- **Suggested Next Step**：Kevin 主动说"上次你的 PRD 还有 3 个 block 缺证据，你新上传的访谈笔记里有相关内容"
- **Artifact 重新打开**：Kevin 展示"上次的工作进展"，而不是空白界面
- **Diff 决策记忆**：Kevin 提示"你上次拒绝了类似的修改建议，这次是否也跳过？"

这些时刻不依赖 Skill Forge 完整实现，只需要 Artifact + Materials + Audit 数据可读。它们是 MVP 中让用户感知"Kevin 记得我"的最小路径。

### 6.4 UX/IA 基线

UX/IA 以 `../kevin2-ia-ux-v2.md` 为基线，不在本 PRD 中重新设计。
