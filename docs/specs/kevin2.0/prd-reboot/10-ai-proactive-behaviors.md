# Kevin 2.0 AI 主动行为规格

> 状态：Draft  
> 第一性底层：`./00-first-principles.md`（Q2 认知循环分工、Q5 飞轮）  
> 关联：`./11-first-encounter-spec.md`（第一次见面是主动行为的起点）/ `./07-skill-forge-productization.md`（后置沉淀的下游）/ `./05-ux-ia-alignment.md`（Suggested Next Step 生成逻辑）  
> 作用：把"前置辅助"和"后置沉淀"从第一性原理中的抽象原则，落地为可工程实现的行为目录——每一个行为都有触发条件、内容结构、表达模板和边界约束

## 0. 为什么需要这份文档

Kevin 的核心价值承诺（"越用越聪明"、"记得你的判断"、"主动而不打扰"）最终都要落到**具体的行为时刻**——Kevin 在哪个节点出现，说什么话，采集什么信号，沉淀什么到哪里。

如果这些行为没有被规格化，会发生两种失败：

1. **工程侧失败**：没有明确的触发器和内容结构，实现者各自解读，产品体验不一致
2. **产品侧失败**：行为太多变成打扰，行为太少让"主动"承诺落空；没有后置沉淀则飞轮的方法引擎永远不转

本文定义 Kevin 主动行为的**完整规格**：设计原则 → **连接/处理行为目录** → 前置辅助行为目录 → 后置沉淀行为目录 → **执行/监控行为目录** → 飞轮闭环接口 → MVP 最小集 → 评测方法。

### 根本认知：四个阶段的依赖关系

这四类行为不是平行的功能模块，而是有依赖顺序：

```
连接/处理行为（基础设施）
  ↓ 提供可信的数据基础
前置辅助行为（认知环境准备）
  ↓ 让洞察发生
用户判断（人类主权，不在本文范围）
  ↓ 产生可采集的判断信号
后置沉淀行为（私有方法学习）
  ↓ 反馈给前置辅助，改进下一次
  ↓ 同时生成执行意图
执行/监控行为（意图落地 + 结果感知）
  ↓ 完成闭环，产生下一轮认知输入
```

**关键推论**：
- 连接/处理是**数据管道**——它做不好，前置辅助就没有可信素材，"Kevin 发现了关联"就是空话
- 执行/监控是**信任锚点**——它做不好，用户无法验证 Kevin 的执行是否准确，整个闭环断裂
- 两者都是 AI 主导（不依赖用户判断），因此也是 Kevin 最应该做到**零缺陷**的阶段

## 1. 设计原则

### 1.1 主动边界：在自己主权地带主动，不越界替用户决策

认知循环的分工决定了 Kevin 的主动边界：

| 阶段 | AI 角色 | Kevin 的行为权限 |
|---|---|---|
| 连接 / 获取 / 处理 | **AI 主导** | 自主执行：自动 Material 化、定时索引、连接器监控。用户不需要为每次文件读取或数据刷新审批 |
| 洞察的**前置辅助** | 为人服务 | 主动呈现信息，**不替用户形成结论**；触发需满足三必要条件（见 §1.2） |
| 认知迭代 / 决策 | 人类主权 | **不介入**——Kevin 可以提供素材，不可以说"你应该这样决定" |
| 执行 | **AI 主导（受控）** | 必须经过 Sign-off；执行后主动报告结果闭环，不静默 |
| 监控 | **AI 主导** | 自主持续监控；主动报告变化——变化是事实描述，不附加判断 |

违反这个边界的 Kevin 行为，无论出发点多好，都是**越权**。越权行为长期损害用户信任。

### 1.2 主动的必要条件（三选三）

Kevin 主动出现，必须同时满足：

1. **有真实观察**：Kevin 看到的是真实信息（来自材料、数据、操作记录），不是基于假设的"可能"
2. **有具体指向**：Kevin 能引用具体的文件 / 字段 / 数据 / 历史决策，不是泛化建议
3. **有时机价值**：现在出现比等用户主动问更有价值（如：用户刚上传了新材料，刚完成了一次决策，刚打开了一个沉寂已久的项目）

三个条件缺一不可。缺少任何一个，Kevin 应该保持沉默。

### 1.3 表达语气规范

Kevin 的主动表达必须遵循一套稳定的语气系统——这是 Kevin 性格（严谨 / 主动 / 直接 / 透明 / 尊重主权）的语言落地。

| 语气维度 | 规范 | 反例 |
|---|---|---|
| **确定性匹配** | 表达的确定性必须与实际信心匹配；中等信心用"可能"、"我注意到" | ❌ 50% 信心说"你应该" |
| **主动提问，不主动结论** | 以问题结尾，把判断权交回用户 | ❌ "这个 PRD 的 Problem block 需要修改" |
| **具体，不泛化** | 引用文件名、字段名、数值；不说"有些内容" | ❌ "我看到一些值得注意的内容" |
| **不过度道歉** | 说错时简短纠正，继续；不说"非常抱歉" | ❌ "非常抱歉给您带来了困扰" |
| **不施压** | 建议可以被拒绝；没有"你还没做 X"的催促感 | ❌ "你已经 3 天没有更新 Problem block 了" |
| **透明依据** | 说清楚"我为什么这么认为"——引用来源 | ❌ 没有依据的结论 |

### 1.4 频率上限（防打扰规则）

| 范围 | 上限 |
|---|---|
| 每次会话 | 主动介入 ≤ 3 次（用户主动触发的交互不算） |
| 同类型主动行为 | 同一类型在同一会话内最多出现 1 次 |
| 被拒绝的建议 | 本次会话内不再重复 |
| D3 目录变化提示 | 每次会话最多 1 次 |
| 后置沉淀提议（保存 Skill）| 在用户完成一个完整认知循环后才触发，不提前 |

**频率上限不是目标上限——大多数情况下 Kevin 只需要一次准确的主动介入。**

### 1.5 用户控制权

用户可以：

- 在 Settings 关闭全局主动行为（进入"仅响应"模式）
- 针对特定行为类型静音（如"不再提示 Evidence 缺失"）
- 调整频率偏好（"少一点"/ "正好" / "多一点"三档）

这些偏好本身会被后置沉淀机制记录，并影响后续主动行为的触发阈值。

## 2. 连接/处理阶段行为目录（AI 主导）

这个阶段是 Kevin 的**数据基础设施层**。不同于后续阶段的"主动介入"，这里的行为绝大多数是**静默自主执行**——用户感知到的不是"Kevin 在做什么"，而是"Kevin 的结果总是最新的、可信的、覆盖全面的"。

**设计哲学**：连接/处理阶段的行为失败，表现为沉默——用户不会看到报错，只会在洞察阶段发现素材陈旧或缺失。因此这个阶段的质量标准是**零降级不通知**：任何影响数据质量的降级，必须在 Workspace Home 的 Connectors Status 卡片上可见。

---

### 2.1 本地文件相关

#### [B-C-001] 目录变化自动 Material 化

**性质**：静默自主执行，不打扰用户。

**触发器**：
- File watcher 检测到 Workspace 目录内有文件新增 / 修改 / 删除

**执行动作**：

| 文件事件 | Kevin 动作 |
|---|---|
| 新增可解析文件 | 自动触发增量索引（向量化，不调 LLM）；更新 `directory_overview.total_files`；标记"待深度分析" |
| 新增不可解析文件 | 记录文件元数据（名称、大小、类型）；不索引内容；更新 `directory_overview.file_type_distribution` |
| 文件修改 | 触发该文件的重新索引；更新关联 EvidenceRef 的"可能过期"标记 |
| 文件删除 | 将对应 Material 标记 `stale`；通知引用该 Material 的 Artifact block 的 `support_level` 降级 |

**成本控制**：
- 增量索引只做向量化（嵌入模型，极低成本），不调 LLM
- LLM 推理（跨文件关联 / key_finding 更新）**批量延迟处理**——在 D3 阈值触发时或用户启动会话时批量执行，不逐文件调用
- 同一文件 5 分钟内多次修改（如频繁保存）去重处理，只触发一次索引

**与 D2/D3 的关系**：
```
文件变化 → B-C-001 自动 Material 化（立即，静默）
                    ↓
         变化累积达到 D3 阈值
                    ↓
         主动提示用户（B-P-003 追加发现 / D3 通知）
```

**降级通知**：File watcher 失效时（系统睡眠后未恢复 / 权限变更），在 Connectors Status 卡片显示"本地文件同步已暂停 [重新连接]"。

---

#### [B-C-002] 被删除或移走的 Material 影响通知

**性质**：条件触发，低频；需要用户感知，但不紧迫。

**触发器**：B-C-001 检测到文件删除 / 移动，**且**该文件是某个 Artifact block 的 EvidenceRef 来源

**行为**：在 Workspace Home Suggested Next Step（或 P0 通知）出现一条：

```
[文件名] 已从目录中删除。
以下内容引用了它：[Artifact 名] · [block 名]（[EvidenceRef 片段]）
该引用现在处于待确认状态。[查看影响] [忽略]
```

**边界**：文件只是移动而非删除（新路径在同一 Workspace 目录内）时，自动更新引用路径，不触发通知。

---

### 2.2 数据连接相关

#### [B-C-010] 连接器健康自动监控

**性质**：静默自主执行（定时）+ 异常时主动展示。

**监控策略**：

| 时机 | 动作 |
|---|---|
| 每次 Workspace 打开 | 对所有已配置 Connector 做轻量 ping（不调 LLM，不取数据） |
| 每 4 小时（后台） | 静默复查一次连接状态 |
| ActionRequest 发起前 | 检查目标 Connector 状态；降级则**阻断执行**并明确告知 |

**Connector 状态定义**：

| 状态 | 含义 | Connectors Status 卡片表现 |
|---|---|---|
| `connected` | 正常 | 绿点，静默 |
| `degraded` | 响应慢 / 局部不可达 | 黄点 + "连接不稳定，可能影响速度" |
| `reauth_required` | Token / 授权过期 | 红点 + "[连接器名] 授权已过期，需要重新连接 [重连]" |
| `offline` | 完全不可达 | 灰点 + "[连接器名] 不可达，相关功能暂停" |

**关键原则**：所有 `reauth_required` 和 `offline` 状态**必须在 Workspace Home 可见**，不允许静默失效——静默失效会让用户对 Kevin 产生"不知道发生了什么"的信任危机。

---

#### [B-C-011] 数据材料定时刷新

**性质**：静默自主执行（按调度），用户可配置。

**触发器**：数仓 Material 超过配置的刷新间隔（默认 24 小时）

**执行动作**：
1. 使用相同查询参数重新执行数仓查询
2. 比较新旧结果：
   - 无显著变化 → 更新 `indexed_at` 时间戳，静默
   - 有显著变化（超过 §B-P-020 的阈值） → 触发 B-P-020（关键指标变化提示）
   - 查询失败 → 保留旧数据，标记 `stale`，触发 B-C-010 降级通知

**用户控制**：每个数仓 Material 可单独配置刷新间隔（日 / 周 / 手动），或关闭自动刷新。

---

#### [B-C-012] Artifact 启动时的跨源信息聚合

**性质**：条件触发（用户动作触发），单次 LLM 调用。

**触发器**：
- 用户新建一个 Artifact（选择类型并开始编辑）
- **且** Workspace 内有来自多个不同 Connector 的 Material（不只有本地文件）

**执行动作**：
1. 对所有相关 Material 进行语义检索（向量搜索，不调 LLM）
2. 排序后取 Top-K 相关 Material（K = 5-8）
3. 一次 LLM 调用生成"跨源上下文摘要"：这些材料里哪些与这个 Artifact 最相关，是否有跨源的关联或矛盾

**呈现方式**：在 Artifact 编辑界面的 Inspector 面板"相关材料"区域，按相关性排序展示，标注来源（本地 / 数仓 / 飞书…），不打断主编辑流程。

**成本控制**：只在 Artifact **新建时**触发一次；后续由用户主动刷新，不定时触发。

---

### 2.3 材料质量相关

#### [B-C-020] 大文件自动去噪摘要

**性质**：按需触发（Material 化时），后台执行。

**触发器**：
- 文件大小 > 50KB 的文本文件（PDF / Word / 长 Markdown）完成 Material 化

**执行动作**：后台异步生成结构化摘要：
- 核心主题（2-3 句）
- 关键段落索引（章节 → 内容映射，方便 Evidence 引用定位）
- 不确定或模糊的部分标注

**存储**：摘要附加到 Material 对象，不替代原文；Evidence 引用时同时展示原文片段和摘要上下文。

**MVP 态度**：P2，不阻断主路径——没有摘要时直接用原文，只是 Evidence 引用体验略差。

## 3. 前置辅助行为目录

前置辅助的目标是**准备最好的认知环境，让用户的洞察更容易发生**。

每一个行为条目的格式：

```
[B-P-XXX] 行为名称
触发器：什么事件 + 什么条件 → 触发
内容结构：Kevin 输出的结构
表达模板：具体的话术
边界：什么情况下不触发 / 停止
```

---

### 2.1 材料与认知相关

#### [B-P-001] 材料扫描完成后的关键发现提示

**触发器**：
- 新材料（文件 / 数据查询结果 / 外部文档）完成 Material 化
- **且** Kevin 识别到该材料与现有 Artifact 或 directory_cognition 有至少一条可引用关联

**内容结构**：
1. 指明材料与哪个现有对象有关联
2. 描述具体的关联点（引用文件名 / 章节 / 字段）
3. 提出一个具体问题（不替用户决定怎么做）

**表达模板**：

```
[材料名] 里有一段内容可能和 [已有 Artifact / cognition 里的具体部分] 有关：
"[引用片段，≤ 50 字]"
要不要把这个补充到 [具体位置]？
```

**边界**：
- 置信度 < 0.7 时不触发（静默记录，等 Tier 2 索引完成后重评估）
- 同一材料的关联提示不超过 2 条（更多关联折叠展示，不逐一打断）
- 用户已打开该 Artifact 正在编辑时不触发（避免干扰焦点）

---

#### [B-P-002] 多材料之间的矛盾或缺口提示

**触发器**：
- Tier 2 后台索引完成
- **且** Kevin 发现两个或更多已 Material 化的文件之间存在数据不一致 / 结论矛盾 / 明显信息缺口

**内容结构**：
1. 指明矛盾 / 缺口的具体来源（文件 A vs 文件 B，或 "X 材料提到了 Y，但没有找到 Z 的对应信息"）
2. 说明影响范围（这个矛盾可能影响哪个决策）
3. 提出行动选项

**表达模板**：

```
我发现 [文件 A] 和 [文件 B] 在 [具体主题] 上有不一致：
- [文件 A] 说：[具体表述]
- [文件 B] 说：[具体表述]
这可能影响 [相关 Artifact / 判断]。要先理清这个，还是继续其他？
```

**边界**：
- 矛盾必须是事实层面的（数据不一致 / 明确结论相反），不是风格差异
- 不主动评判哪个材料"更对"

---

#### [B-P-003] 认知 Material 更新后追加发现提示

**触发器**：
- directory_cognition 完成 Tier 2 索引（后台深度扫描完成）
- **且** 新增了至少一条 `key_finding`

**内容结构**：
1. 说明已完成全量读取
2. 列出新增发现（≤ 3 条，其余折叠）

**表达模板**：

```
我把全部内容读完了，有 [N] 个新发现：
• [发现 1，引用具体文件]
• [发现 2，引用具体文件]
[展开 N-2 条] → 查看全部

要从哪里开始？
```

**边界**：
- 如果 Tier 2 完成时用户不在会话内，下次进入 Workspace 时在 Home 展示，不弹出通知

---

### 2.2 Artifact 编辑相关

#### [B-P-010] 开始编辑 Artifact 时的上下文激活

**触发器**：
- 用户打开一个已有 Artifact（不是新建）
- **且** 该 Artifact 上次更新距今 > 24 小时
- **且** 期间有新材料或外部变化可能影响该 Artifact

**内容结构**：
1. 简短说明上次的工作状态（"上次你在处理 X block，有 Y 个地方标记了 needs-review"）
2. 说明期间发生了什么（"这期间你新上传了 Z 材料"）
3. 提议从哪里接着开始

**表达模板**：

```
欢迎回来。上次 [Artifact 名] 的工作停在：
• [block 名] 还有 [N] 个位置标记了 needs-review
• [有新材料时：] 这期间你上传了 [材料名]，里面有内容可能支撑 [具体 block]

从 needs-review 继续，还是先看新材料？
```

**边界**：
- 距上次更新 < 24 小时时不触发（用户还记得上下文）
- 如果 Artifact 完全干净（无 needs-review 且无新材料），不触发

---

#### [B-P-011] block 缺少 Evidence 的主动提示

**触发器**：
- 用户在某个必填 block 完成内容编辑（失去焦点 or 切换 block）
- **且** 该 block 当前 `support_level` 为 `inferred` 或 `needs-review`
- **且** Kevin 在已有 Material 中能找到至少一条潜在支撑证据

**内容结构**：
1. 指出 block 当前缺乏材料支撑
2. 给出 Kevin 找到的具体候选证据（引用片段 + 来源）
3. 询问是否引用

**表达模板**：

```
[block 名] 目前没有材料引用。
我在 [材料名] 里找到了可能相关的内容：
"[引用片段，≤ 60 字]"（[材料名] · [位置提示]）
要把这个作为 Evidence 引用吗？[引用] [不合适]
```

**边界**：
- 用户已经主动标注了 Evidence 时不触发
- 如果 Kevin 找不到候选证据，只静默标注 `needs-review`，不说"我没找到"（避免干扰编辑节奏）

---

#### [B-P-012] diff 建议携带上下文激活

**触发器**：
- Kevin 生成 diff 建议（由 AI Review 触发）
- **且** 该建议与用户的历史决策有关联（用户曾拒绝过类似建议，或曾明确偏好某种处理方式）

**内容结构**：
1. 正常呈现 diff
2. 在 diff 下方追加：Kevin 的历史观察（"你上次拒绝了类似建议，理由是 X"）
3. 给用户三条路径：接受 / 拒绝 / 说明这次不同

**表达模板**：

```
[正常 diff 内容]

── Kevin 注意到 ──
你上次处理类似的 [block 类型] 时，拒绝了相似的修改，当时的记录是："[历史拒绝原因]"。
这次的情况 [相似 / 有所不同，具体说明]。
[接受] [跳过，理由相同] [这次不同，[说明]]
```

**边界**：
- 只在有历史决策记录时触发（无记录则正常展示 diff）
- 历史记录的置信度 < 0.8 时不引用（静默）

---

#### [B-P-013] 跨 Artifact 一致性提示

**触发器**：
- 用户在 Artifact A 里修改了某个核心字段（如策略方向、关键指标定义）
- **且** Kevin 检测到 Workspace 内另一个 Artifact B 中有依赖相同内容的 block

**内容结构**：
1. 指明跨 Artifact 影响
2. 具体到哪个 Artifact 的哪个 block
3. 询问是否同步

**表达模板**：

```
你在 [Artifact A] 里修改了 [具体内容]。
[Artifact B] 的 [block 名] 里有引用这个内容的表述，现在可能需要同步。
要看一下吗？[查看影响] [暂时跳过]
```

**边界**：
- 只有 EvidenceRef 明确引用关系时触发，不做语义猜测
- 用户跳过后本次会话内不再提示

---

### 2.3 数据与外部信号相关

#### [B-P-020] 关键指标变化提示

**触发器**：
- 数仓 Material 的定时刷新完成（或用户触发查询）
- **且** 查询结果中某个指标值与用户上次查看时有显著变化（变化幅度超过阈值）
- **且** 该指标与当前 Workspace 内某个 Artifact 有 EvidenceRef 关联

**内容结构**：
1. 指明变化的指标（名称 / 值 / 变化幅度 / 时间范围）
2. 指明影响哪个 Artifact 的哪个 block
3. 提出行动选项

**表达模板**：

```
[指标名] 出现了变化：
[上次值] → [当前值]（[时间范围]，变化 [+/-X%]）

这个指标被 [Artifact 名] 的 [block 名] 引用。
要更新引用数据，还是先看一下完整趋势？[更新] [查看趋势] [暂时忽略]
```

**边界**：
- 变化阈值建议起步值：同比 ±15% 或环比 ±10%（可在 Workspace 设置里调整）
- 同一次刷新最多提示 2 个指标变化（其余折叠）
- 用户关闭了某个指标的提示后，不再触发

---

#### [B-P-021] 长期未更新的数据材料提示

**触发器**：
- 用户打开含数据引用的 Artifact
- **且** 该 Artifact 引用的某个数仓 Material 超过 N 天未刷新（默认 7 天）

**内容结构**：
1. 指明哪个 Material 已过时
2. 指明影响的 block

**表达模板**：

```
[Artifact 名] 引用的 [查询名 / 指标名] 数据已有 [N 天] 未更新。
可能影响 [block 名] 里的数据准确性。
要刷新查询吗？[刷新] [暂时跳过]
```

**边界**：
- 不在用户正在编辑 block 时触发（等编辑失焦后）

---

### 2.4 工作节奏相关

#### [B-P-030] Suggested Next Step 刷新提示

这是 Workspace Home 的核心主动行为，规格已在 `05-ux-ia-alignment.md` §4.1.1 详细定义，本文不重复。

关键原则：Suggested Next Step 的生成逻辑消费后置沉淀的产出（历史判断模式、拒绝记录、偏好），是前置辅助 + 后置沉淀**飞轮的可见输出窗口**。

---

#### [B-P-031] 重新进入长期搁置项目的激活

**触发器**：
- 用户打开一个 ≥ 14 天未有任何交互的 Workspace
- **且** Workspace 内有未完成状态的 Artifact 或 pending ActionRequest

**内容结构**：
1. 说明上次工作状态的快照（不是长篇总结，是 3-5 个关键点）
2. 说明期间可能需要关注的变化（材料过期、待处理动作等）
3. 提出重新进入的路径

**表达模板**：

```
距上次在这个项目工作已经 [N 天]。
上次的状态：
• [Artifact 名] 停在 [状态]，[N] 个 needs-review
• [有待处理 Action 时：] [N] 个待确认 Action

期间有一些变化：
• [变化 1]
• [变化 2]

从哪里开始？[继续上次] [看全部状态] [重新认识这个项目]
```

**边界**：
- 每个 Workspace 最多每 7 天触发一次（避免反复打扰）
- 如果用户点击"重新认识这个项目"，触发 directory_cognition 重扫（见 spec 11 §8.3）

## 3. 后置沉淀行为目录

后置沉淀的目标是**精确捕获用户的判断，沉淀私有方法，作为下一次前置辅助的基础**。

后置沉淀是**静默的**——它不打扰用户，但持续采集信号。只有当积累达到一定阈值时，Kevin 才会主动表面化沉淀的结果，邀请用户确认。

---

### 3.1 判断信号采集

| 信号类型 | 触发动作 | 数据记录 | 沉淀潜力 |
|---|---|---|---|
| **diff 接受** | 用户接受 Kevin 的 diff 建议 | 被接受的内容类型、block 类型、修改方向 | 风格偏好、内容判断 |
| **diff 拒绝** | 用户拒绝 diff 建议 | 被拒绝的内容类型、用户的可选原因 | 禁止模式、偏好反向 |
| **diff 修改后接受** | 用户编辑了 Kevin 的建议内容再接受 | 原始建议 vs 最终内容（diff of diff） | **最高价值信号**——体现私有方法 |
| **Evidence 确认** | 用户确认 EvidenceRef 引用正确 | 材料类型、block 类型、证据用法 | 证据标准 |
| **Evidence 拒绝 / 替换** | 用户删除或替换 EvidenceRef | 被替换的证据 vs 新证据 | 用户的证据标准 |
| **手动修改内容** | 用户直接编辑 block 内容（非通过 diff） | 修改前后内容（需用户同意采集） | **高价值信号**——私有表达风格 |
| **主动建议拒绝** | 用户拒绝 Kevin 的主动提示 | 被拒绝的行为类型、上下文 | 不打扰偏好、建议过滤 |
| **ActionRequest 批准** | 用户 Sign-off 一个 ActionRequest | 动作类型、风险等级、前置条件 | 风险容忍度、决策标准 |
| **ActionRequest 拒绝** | 用户拒绝 ActionRequest | 动作类型、拒绝理由 | 不可接受操作 |
| **Chat 纠正** | 用户在 Chat 中说"这不对"、"应该是 X" | 被纠正的内容和正确内容 | 概念理解、领域知识 |

### 3.2 后置沉淀产出类型

采集的信号经过聚合，产出以下类型的沉淀对象：

#### 3.2.1 偏好记录（Preference Record）

**定义**：用户在重复场景下稳定表现出的倾向——不是规则，是统计趋势。

**形成条件**：同类信号在 ≥ 3 次独立操作中出现。

**示例**：
- "你在 Risks block 里倾向于把风险写成决策导向的条目（而不是描述性列表）"（来自 3 次 diff 修改信号）
- "你通常在 Evidence 引用时偏好引用原始数据而不是分析结论"（来自 5 次 Evidence 替换信号）

**存储**：非持久化 Skill——存在 Workspace 层的 `preference_store` 中，直接影响前置辅助的生成提示。

**用户交互**：不需要用户确认即生效；但会在 Kevin 使用偏好时透明标注："（按照你在这个项目里的习惯）"，并提供"重置此偏好"按钮。

#### 3.2.2 判断框架草案（Skill Draft）

**定义**：用户在一类工作中稳定应用的评估框架或决策方法——比偏好更高层，是"私有方法"的核心载体。

**形成条件**：
- 相关信号在 ≥ 5 次独立操作中出现
- **且** Kevin 能识别出这些信号背后有一个可描述的逻辑（不只是"喜欢 A 不喜欢 B"）

**示例**：
- "你分析留存数据时惯用同期群对比，并优先关注首日留存而不是 D7/D30"（来自 5 次数据引用信号）
- "你评估功能风险时习惯先问'这个能回滚吗'，再问'影响范围多大'"（来自多次 diff 修改 + ActionRequest 决策信号）

**与偏好记录的区别**：偏好记录是"倾向"（A > B），判断框架是"方法"（先 X 再 Y，用 Z 标准评估）。

**存储**：形成后主动向用户呈现为 Skill Draft（见 §3.3 沉淀可见时刻），经用户确认后升级为正式 Skill。

#### 3.2.3 决策记忆（Decision Memory）

**定义**：用户在特定情境下做出的一次性判断，不一定是稳定的规律，但在类似情境下值得参考。

**形成条件**：
- 一次完整的 ActionRequest 批准 / 拒绝
- 一次 Chat 纠正，且用户明确说明了原因
- 一次用户对 diff 建议的"修改后接受"，且修改幅度较大

**示例**：
- "2026-05-01，你决定把 Feishu 投影延迟到工程对齐完成后"（来自一次 ActionRequest 拒绝）
- "2026-04-22，你把原来的'用户需求描述'改写为'用户问题陈述'——这次改写的方向是去掉方案导向"（来自一次 diff 修改信号）

**存储**：存在 `directory_cognition` 的 `key_decisions` 字段（需在 doc 02 §2.2.2 后续版本补充），也作为 Evidence 可被 Artifact 引用。

### 3.3 沉淀可见时刻

后置沉淀是静默的，但在以下时刻**主动表面化**，让用户看到积累正在发生：

#### [B-S-001] 偏好确认提示

**触发条件**：同类偏好信号达到 3 次阈值时。

**表达模板**：

```
Kevin 注意到你在 [工作场景] 时，倾向于 [具体偏好描述]。
这个偏好已在这个项目里出现 3 次。
要设为这个项目的默认方式吗？[设为默认] [这次是例外] [不用记录]
```

**用户选择说明**：
- **设为默认**：写入 Preference Store，后续前置辅助自动应用
- **这次是例外**：本次不记录，但不清除之前的信号
- **不用记录**：清除此类偏好信号，Kevin 不再尝试形成此偏好

---

#### [B-S-002] 判断框架草案提议

**触发条件**：判断框架草案形成条件达到（≥ 5 次信号 + 可识别逻辑）。

**表达模板**：

```
Kevin 观察到你处理 [一类工作] 时，有一个稳定的方式：
[判断框架的 2-3 句描述，用用户自己的语言表达]

这个模式在你最近 [N] 次工作中都出现了。
要把这个保存为一个可复用的方法吗？[预览] [保存为 Skill] [不用保存]
```

**预览内容**（点击"预览"后展示）：

```
── Skill 草案预览 ──
名称：[Kevin 建议的命名]（可编辑）
适用于：[工作类型描述]
方法：
  1. [步骤 1]
  2. [步骤 2]
  3. ...
来源：[N] 次工作观察（可查看依据）
```

用户可以编辑名称、调整描述，然后保存。也可以拒绝——拒绝时询问"是方法描述不准确，还是不需要保存？"以便修正信号。

---

#### [B-S-003] 完整认知循环闭环时刻

**触发条件**：用户完成一个完整认知循环（Materials → Artifact → ActionRequest approved / executed → 结果可见）。

这是**飞轮可见性的核心时刻**——用户第一次看到"一次工作 = 一次积累"。

**表达模板**：

```
你完成了从材料到制品到执行的完整工作循环。

这次循环里 Kevin 积累到：
• [N] 个材料引用关系
• [M] 个判断偏好信号
• [如有：1 个方法草案待确认]

这些积累将作为这个项目下一次工作的基础。[查看全部积累] [继续工作]
```

**边界**：每个 Workspace 只触发一次（首次完整循环后）；后续完整循环静默记录。

---

#### [B-S-004] 同类工作提速通知

**触发条件**：用户完成同类工作的第 3 次（或之后每隔 3 次），且 Kevin 检测到这次耗时 / 修改轮数明显少于第 1 次。

**表达模板**：

```
这是你第 [N] 次完成 [工作类型]。
Kevin 注意到这次 [比上次少修改了 M 轮 / 快了约 T 分钟]。
当前积累的偏好和方法已在这次工作中应用。[查看应用了哪些]
```

## 4. 执行/监控阶段行为目录（AI 主导）

这个阶段是 Kevin 的**信任锚点**。执行是用户判断落地的最后一公里；监控是让 Kevin 真正做到"不睡觉的工作伙伴"。

**设计哲学**：执行的最大风险不是失败，而是**静默失败**——Kevin 执行了但用户不知道执行到哪里，或者执行错了但没有反馈。监控的最大风险不是漏报，而是**噪声打扰**——每次无关变化都通知用户，信噪比崩溃后用户开始忽略所有通知。

因此这个阶段的两个核心设计约束：
1. **执行必须有闭环反馈**，结果可见，不允许静默
2. **监控必须用户配置触发条件**，Kevin 不自主决定什么值得汇报

---

### 4.1 执行阶段

#### [B-E-001] 执行结果确认闭环

**性质**：执行后自动触发，无例外。

**触发器**：任意 ActionRequest 完成执行（成功或失败）

**反馈内容与呈现位置**：

| 执行类型 | 反馈内容 | 呈现位置 |
|---|---|---|
| 文件写出（export / write to library） | "已写入 `[路径]`，[文件大小]，[打开文件]" | Artifact 底部状态栏 + Audit 条目 |
| 外部投影创建（如飞书文档） | "已创建 [平台] 文档：`[标题]` [打开链接]；已投影 [N] 个章节" | Action Panel 确认卡 + Inspector 外部链接 + Audit 条目 |
| 数仓查询执行 | "查询完成：[N] 行 · [N] 列，已 Material 化为 `[材料名]` [查看]" | Chat 回复 + Material List |
| ActionRequest 失败 | "执行失败：[具体原因]。[重试] [查看详情]" | Action Panel 错误卡（红色） + Audit 条目 |

**为什么是 P0**：确认闭环不需要额外 LLM 调用——是纯状态机逻辑（执行 → 读取结果 → 格式化展示）。工程成本极低，但消除了"Kevin 做了什么我完全不知道"的信任黑洞。

**Audit 写入**：每次执行（成功或失败）必须写入 AuditEntry，见 `08-connector-capability-governed-action.md` §8。

---

#### [B-E-002] 执行后外部链接状态感知

**性质**：执行后定时轻量检查（不调 LLM）。

**触发器**：外部投影（飞书文档创建）成功 24 小时后

**执行动作**：通过外部 Connector 读取文档元数据（如最后修改时间、是否存在），验证链接仍然有效

**呈现**：
- 文档仍然存在 → 静默
- 文档被删除 → 在 Inspector 外部链接区域标记 `⚠️ 文档已被删除`

**成本控制**：只检查文档元数据（轻量 API 调用），不读取文档内容；每个外部链接只检查一次（24 小时后触发一次，之后静默）。

**MVP 态度**：P1——飞书文档创建是 MVP 主路径，这个检查可以简单实现。

---

### 4.2 监控阶段

#### [B-E-010] 指标守望（用户配置触发）

**性质**：用户显式配置 + 定时自主执行 + 条件触发主动提示。

**设计理由**：这是"AI 不睡觉"这个价值承诺的最直接体现。对于有数据监控需求的用户，一个运转良好的指标守望，可能比所有其他功能加在一起提供更多日常价值。

**配置模型**：用户在 Workspace 设置里配置一条或多条 Watch：

```
Watch 配置：
  名称：留存警报
  指标：D1 留存率（绑定 Metric Dictionary 中的 metric_id）
  查询：[关联的数仓查询 Material]
  触发条件：同比下降 > 10%  |  环比下降 > 5%  |  绝对值低于 30%
  关联：[增长 PRD] · [Goals block]（可选）
  通知频率：每天最多 1 次
```

**执行流程**：

```
定时触发（用户配置的检查周期，默认每天）
  → 执行关联数仓查询（不调 LLM）
  → 计算是否满足触发条件（纯计算）
  → 条件未满足 → 静默记录，更新"上次检查时间"
  → 条件满足 → 一次 LLM 调用生成提示话术 → 主动通知
```

**触发时的表达模板**（参考 B-P-020）：

```
[Watch 名称] 触发了：
[指标名] [当前值] → [上次值]（[变化幅度]，[时间范围]）
触发条件：[用户配置的条件描述]

[如有关联 Artifact：] 这个指标被 [Artifact 名] 的 [block 名] 引用。
[查看数据] [更新 Artifact 引用] [暂时忽略]
```

**成本控制关键**：
- 定时查询成本极低（SQL 执行，无 LLM）
- 只有触发时才调一次 LLM（生成话术）
- 未触发时零 LLM 成本

**MVP 约束**：依赖 Data Warehouse Connector；MVP 内 Watch 数量上限建议为每个 Workspace ≤ 5 条（避免定时任务爆炸）。

---

#### [B-E-011] 关键 Material 过期主动提示

**性质**：条件触发，在 Workspace 打开时评估。

**触发器**：
- 用户打开含数据引用的 Artifact
- **且** 该 Artifact 中某个 block 的 EvidenceRef 来源 Material 的 `indexed_at` 超过 N 天（默认 7 天）且未自动刷新

**表达模板**：

```
[Artifact 名] 引用的 [材料名 / 查询名] 数据已有 [N 天] 未更新。
可能影响 [block 名] 的数据准确性。
[刷新查询] [查看当前数据] [暂时跳过]
```

**边界**：
- B-C-011 数据材料定时刷新已完成时，不触发本行为（避免重复）
- 用户已选择"手动刷新"模式的 Material 不触发
- 每次会话最多提示 1 次（打开 Artifact 时）

---

#### [B-E-012] 持续工作状态汇报（会话结束时）

**性质**：会话结束时触发（用户关闭 Kevin 或长时间无操作），一次性汇总。

**设计理由**：让用户在离开 Kevin 时知道"Kevin 在我不用它的时候也没有停"——这是"不睡觉的工作伙伴"感知的具体载体。

**触发器**：用户会话结束（窗口关闭 / 超过 30 分钟无操作）

**内容**：本次会话期间，后台自主完成的事情的简短摘要：

```
这次工作期间，Kevin 在后台完成了：
• 索引 [N] 个新文件
• 刷新 [N] 个数据查询
• 检测到 [N] 个指标变化（[查看]）

下次打开时，Kevin 会继续监控这个项目。
```

**边界**：
- 后台无任何新活动时，不展示（不说废话）
- 会话时长 < 5 分钟时不触发（刚打开就关了，没有"持续工作"的感知价值）

---

## 5. 飞轮闭环：前置辅助消费后置沉淀

这是整个行为体系的核心逻辑——前置辅助和后置沉淀不是两个独立的功能，是同一个飞轮的两端。

```
后置沉淀产出
  偏好记录 ────────────────→ 前置辅助输入
  判断框架（Skill）─────────→ 前置辅助输入
  决策记忆 ────────────────→ 前置辅助输入
                                 ↓
                    Kevin 生成"按你的方式准备好了信息"
                                 ↓
                         用户更快形成洞察
                                 ↓
                    用户做出判断（被后置沉淀捕获）
                                 ↑
                         ─────────────────
```

### 4.1 消费接口

| 后置沉淀产出 | 消费方（前置辅助） | 消费时机 |
|---|---|---|
| 偏好记录（Preference Record） | AI Review diff 生成 | 每次生成 diff 时，按偏好调整建议内容和风格 |
| 偏好记录 | B-P-011（Evidence 提示） | 匹配用户偏好的证据类型优先推荐 |
| 判断框架（Skill） | Artifact 生成（材料 → 内容） | 按用户判断框架组织生成内容结构 |
| 判断框架 | B-P-001（材料关联提示） | 按框架识别什么材料值得提示 |
| 决策记忆 | B-P-012（diff 历史上下文） | 在相似情境下引用历史决策 |
| 决策记忆 | Suggested Next Step | 预判用户可能优先做的下一步 |
| 全部三类 | directory_cognition 演进 | 更新认知文件中的 `key_findings` 和隐含判断倾向 |

### 4.2 飞轮质量评估

飞轮是否在转动，可以通过以下指标监测：

| 指标 | 健康值 | 警告值 |
|---|---|---|
| 偏好记录形成率 | > 1 条 / 5 次工作循环 | 0 条 / 20 次工作循环 |
| 判断框架草案形成率 | > 1 个 / 完整项目周期 | 0 个 / 完整项目周期 |
| 前置辅助被接受率（不被拒绝） | > 60% | < 30% |
| diff 修改后接受比例（vs 直接接受） | > 20%（说明 Kevin 在学习） | < 5%（说明 Kevin 没在学）|
| Suggested Next Step 被执行率 | > 40% | < 15% |

## 7. MVP 主动行为最小集

MVP 不必实现全部行为，但以下是**不实现就无法支撑产品承诺**的最小集。

### 7.1 连接/处理 MVP 集（数据基础，P0）

| 行为 | 重要性理由 | 成本 |
|---|---|---|
| **B-C-001** 目录变化自动 Material 化 | 数据管道基础——没有这个，前置辅助无可信素材 | 低（增量向量化，不调 LLM） |
| **B-C-010** 连接器健康监控 | 防止静默失效——失效不通知是最大信任杀手 | 极低（定时 ping） |
| **B-C-011** 数据材料定时刷新 | 保证数据不陈旧，Artifact 引用的数据有意义 | 低（定时 SQL，不调 LLM） |

### 7.2 前置辅助 MVP 集

| 行为 | 重要性理由 | 成本 |
|---|---|---|
| **B-P-001** 材料关联提示 | 第一次见面之后的第一个"AI 发现了关联"——信任建立的关键 | 中（索引触发后一次 LLM） |
| **B-P-010** Artifact 上下文激活 | "Kevin 记得你的工作"感知，成本低价值高 | 低（读状态，不调 LLM） |
| **B-P-011** Evidence 缺失提示 | Artifact 质量的最小门卫 | 低（向量搜索 + 条件判断） |
| **B-P-031** 搁置项目重新激活 | 防止"Kevin 只有第一次好用" | 低（读时间差 + 状态汇总） |

### 7.3 执行/监控 MVP 集（信任锚点，P0）

| 行为 | 重要性理由 | 成本 |
|---|---|---|
| **B-E-001** 执行结果确认闭环 | 每次执行后的结果反馈——消除"不知道发生了什么"的信任黑洞 | 极低（纯状态机，不调 LLM） |
| **B-E-011** 关键 Material 过期提示 | 防止用户基于陈旧数据做决策 | 低（时间戳比较） |

### 7.4 后置沉淀 MVP 集

| 行为 | 重要性理由 | 成本 |
|---|---|---|
| diff 接受/拒绝/修改信号采集 | 飞轮方法引擎的数据基础，不采集则飞轮永不转 | 极低（事件记录） |
| **B-S-001** 偏好确认提示 | 飞轮可见性的最低门槛 | 低（规则判断 + 模板生成） |
| **B-S-003** 完整循环闭环时刻 | 产品故事的关键一幕，一次性触发 | 低（状态检测 + 模板生成） |

### 7.5 MVP 内不做的行为

| 行为 | 推迟理由 |
|---|---|
| B-C-012 跨源信息聚合 | 需要多 Connector 同时接入，MVP 以本地文件为主 |
| B-C-020 大文件去噪摘要 | 改善体验但不阻断主路径，P2 |
| B-P-002 多材料矛盾提示 | 精度要求高，MVP 期间宁可不做也不做错 |
| B-P-012 diff 历史上下文 | 需要足够历史数据，冷启动期无效 |
| B-P-013 跨 Artifact 一致性 | MVP 期间 Artifact 数量少，触发概率低 |
| B-E-002 外部链接状态感知 | 飞书 API 依赖，可后置 |
| B-E-010 指标守望 | 依赖 Data Warehouse Connector，非 MVP 首要路径 |
| B-E-012 会话结束状态汇报 | 锦上添花，不是最小集 |
| B-S-002 判断框架草案提议 | 需要 ≥ 5 次工作循环，MVP 期间触发概率低 |
| B-S-004 同类工作提速通知 | 需要纵向数据，MVP 期间用户基础尚浅 |

## 9. 开放问题

### 8.1 信号采集的用户授权

手动修改 block 内容（diff of human edit）是最高价值的后置沉淀信号，但采集前后内容需要用户同意。同意的时机和颗粒度需要设计：

- 首次启用 Kevin 时一次性全局同意？
- 还是每次捕获时询问？
- 可以选择某些 Artifact / Workspace 内不采集？

### 8.2 偏好记录和判断框架的跨 Workspace 共享

MVP 内偏好和框架只在 Workspace 层有效。未来：

- 用户级共享（所有 Workspace 都应用）是否合适？
- 团队级共享（同一 Space 内共享某些判断框架）的冲突处理？

### 8.3 错误沉淀的修正

如果用户某次判断是"特例"但 Kevin 误判为稳定偏好，如何让用户发现并修正？

- 偏好记录是否有"影响来源"的追溯入口？
- 用户说"这次建议用错了我的习惯"时，Kevin 如何响应？

### 8.4 私有方法的敏感性

判断框架可能包含用户的核心竞争方法，需要确保：

- Skill 草案不被意外推送到团队共享层
- 团队共享时需要用户显式确认"这个方法可以被团队看到"
