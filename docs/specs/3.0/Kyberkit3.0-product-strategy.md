## 1. 产品定位（Product Positioning）

### 1.1 核心命题

**企业数字员工的运行时与能力网络** —— KyberKit 让每个用户拥有可被信任、可被审计、可自我进化的私人数字员工；这些数字员工在用户既有权限范围内、按任务限定权限获取上下文并调用工具，**按节奏自主运行**，并将完成任务的方法沉淀为团队和组织可复用的能力资产。

3.0-4.0 的中心叙事从「更聪明的聊天助手」升级为：

> KyberKit 是企业权限环境下的委托式数字员工平台：Agent 以用户身份、在任务限定权限内获取上下文、调用工具，**按周期 / 触发 / 即时三种合约持续运行**，并将工作能力自我进化为组织可复用资产。

这意味着 KyberKit 的竞争焦点不是单纯模型能力，而是企业 Agent 真正难以规模化的**五件事**：

1. **上下文访问**：在企业既有权限边界内，跨结构化（业务数据 / 看板）与非结构化（文档 / 日志 / IM）数据源获取上下文。
2. **工具执行**：在合约限定的工具集合内安全执行，含可逆 / 高风险 / 禁止动作分级。
3. **权限约束**：UserPerm ∩ TaskPerm ∩ PolicyConstraint 的有效权限计算与可审计裁决。
4. **节奏运行**：支持周期性（每天凌晨）、触发式（告警 / 客户咨询）、即时式（用户主动）三种任务合约，让 Agent 不只是「问一句答一句」，而是按运营节奏陪跑。
5. **能力沉淀**：把工作中产生的方法以技能 / 偏好 / 工作流形式沉淀，形成组织可复用资产。

「节奏运行」是 3.0 在 4.5/5.0 之前不被 Cursor / Cowork / 各类 IDE-Agent 蚕食的关键差异 —— 它把 Agent 从「副驾」拉到「值班同事」位置。

### 1.2 个人价值 vs 企业价值：Federation 模型

企业与个人诉求看似矛盾（隐私 vs 共建），通过 **分层所有权（Federation）** 可以同时满足：

```
┌─────────────────────────────────────────────────────────┐
│ 个人 Agent（身份层 — 始终私有，不可被组织读取）           │
│  • KK.md（人设 / 价值观）                               │
│  • MEMORY.md（个人工作记忆）                             │
│  • 对话历史 / Trajectory                                │
├─────────────────────────────────────────────────────────┤
│ 技能层（能力层 — 可主动「晋升」，Copy-on-Write）          │
│  • private Skills → 用户 promote → team / org Skills    │
│  • ← fork ← Org Skill Library                           │
├─────────────────────────────────────────────────────────┤
│ 组织上下文层（只读注入，不上传个人原始对话）               │
│  • 企业知识库（Obsidian / 飞书等外部编辑，注入上下文）      │
│  • Org Skill Library（团队共建、排序 / 推荐）             │
└─────────────────────────────────────────────────────────┘
```

**原则**：


| 维度                   | 所有权  | 说明                                 |
| -------------------- | ---- | ---------------------------------- |
| **Agent Identity**   | 个人独占 | 人设、记忆、会话轨迹不离开用户边界（除非用户显式导出）        |
| **Agent Capability** | 可共享  | 技能以 Fork / Promote 方式复制到团队库，源码级可追溯 |
| **Org Context**      | 组织可读 | 知识库 MD、团队技能元数据用于注入上下文，非个性化隐私       |


### 1.3 委托式数字员工模型（Delegated Digital Employee）

个人助理 Agent 的身份不是独立主体，而是**代理用户完成特定任务的数字员工**。Agent 不应获得超出用户的企业权限；它的有效权限来自用户已有权限与当前任务所需权限的交集，并受到组织策略约束：

```text
EffectiveAgentPermission = UserPermission ∩ TaskPermission ∩ PolicyConstraint
```

- **UserPermission**：用户在企业系统中已经拥有的权限，由原系统决定，KyberKit 不绕过。
- **TaskPermission**：完成当前任务所需的最小权限集合，由计划阶段生成，并由用户或策略确认。任务可以是即时（Ad-hoc）、周期（Recurring）、触发（Triggered）三种合约形态之一（见 §4.4）。
- **PolicyConstraint**：企业管理员设定的额外安全策略，例如禁止导出、禁止跨部门传播、禁止自动发送外部消息。

> **3.0 简化版（PolicyConstraint）**：3.0 单用户阶段尚无 Org Admin / Policy Engine，PolicyConstraint 由 KyberKit 提供「**保守 / 平衡 / 开发**」三档内置默认策略包，并允许用户在 KK 配置中覆盖。
>
> - **3.0 公司内 MVP 阶段默认 = 「开发」档**（见 §11.1 决议）：宽松的工具白名单、低审批密度，便于团队内部敢用、敢试，快速验证 LearningLoop / Workspace / Connector 的真实价值。
> - **3.5 起或外发企业客户前**：默认基线上调为「平衡」；面向企业销售时默认「保守」。
> - 3.5 起引入团队级 allowlist，4.0 升级为企业 Org Policy Engine。
>
> 这样 §6 中 P0 的权限交集公式在 3.0 即可被验收，而非「有公式无承载」。

这一模型区别于 OpenClaw 式的个人设备授权：

```text
OpenClaw: AgentPermission ≈ DevicePermission
KyberKit: AgentPermission = UserPermission ∩ TaskPermission ∩ PolicyConstraint
```

KyberKit 的产品承诺是：**代理而非越权，委托而非接管**。

### 1.4 与 Memo Q1 的对齐

Memo 中「企业内部协作智能体集群平台的单用户深度版本」与本节的 Federation 叙事一致：**3.0 先把个人 Agent 做深**，同时预埋多用户与技能流通接口；**3.5** 引入小团队技能协作桥梁；**4.0** 完成企业级集群与治理。

## 2. 目标客户与价值主张（ICP & Value Proposition）

### 2.1 核心目标客户（3.0）

- **企业内「效率先锋」**：运营、数据分析、产品经理、职能专家等 —— **非纯开发者**，但有自动化与文档工作流诉求。
- **次要受众**：开发者（CLI/TUI）、内部效率负责人（评估 PoC）。

### 2.2 各版本 ICP 演进


| 版本      | ICP                    | 核心价值                                |
| ------- | ---------------------- | ----------------------------------- |
| **3.0** | 单人深度用户（含企业内先锋）         | 私有、可进化的专属 Agent；可靠输出；Workspace 异步任务 |
| **3.5** | 2–10 人高效能小组            | Team Skill Hub；技能 CoW 分享；最小化多用户 PoC |
| **4.0** | IT / 安全 / 业务部门 + 规模化员工 | 多租户、RBAC、技能市场、协作知识库、Org Admin       |


### 2.3 价值主张三层次

1. **个人效率**：更少重复劳动，记忆与技能随使用变强。
2. **团队协同**：同一方法论以技能副本形式流转，独立演化不互相覆盖。
3. **组织资产**：可审计的技能库与知识注入，沉淀为可复用的数字资产。

### 2.4 企业场景的核心痛点

企业 Agent 难以规模化的根本原因并非模型不够智能，而是 Agent 无法在合规边界内获得足够**做事和做成事**的上下文数据与工具权限。

- **个人场景**：OpenClaw 一类产品通过本地部署和助理人设，让用户愿意让渡更多隐私数据和设备权限。
- **企业场景**：数据权限、应用权限、信息安全政策更复杂；完成任务所需的上下文与工具链更多，潜在释放的价值也更大。
- **KyberKit 的突破点**：不绕过企业权限体系，而是将权限、任务、上下文、工具和审计产品化，使 Agent 能在企业既有权限边界内安全地做成任务。

## 3. 竞争分析（Competitive Landscape）

### 3.1 竞品矩阵（摘要）


| 竞品                                                             | 定位 / 强项               | 相对 KyberKit 的弱点或差异                                                       |
| -------------------------------------------------------------- | --------------------- | ------------------------------------------------------------------------ |
| **Claude Cowork**（Anthropic，面向知识工作，桌面 App，插件与 Google/Office 等） | 品牌、分发、企业套餐；非技术用户友好    | 能力偏**静态插件**；**无**用户间技能自生成与共享叙事；**无**中国企业 IM 网关（企业微信 / 飞书）；订阅绑定 Anthropic |
| **Claude Code**                                                | 开发者 CLI、Memory + 工具成熟 | 开发者场景；无企业微信 / 飞书；无 LearningLoop 式自进化与团队技能沉淀                              |
| **Cursor Background Agent**                                    | IDE 内异步执行、体验好         | 绑定代码与 IDE；跨域持久记忆与组织技能库非主线                                                |
| **OpenClaw**（开源，本地优先，多渠道 IM，ClawdHub）                          | 开源生态、多模型、个人持久上下文      | 技能多依赖**手动安装**；**无**自生成 / LearningLoop；**无**中国企业协作网关；shell 级能力带来安全运营负担    |
| **Glean / Moveworks**                                          | 企业搜索 / IT 服务、销售渠道     | 偏检索与既定工作流；高实施成本；非「个人私有 + 自进化」路线                                          |
| **Dify / Coze**                                                | 低代码编排                 | 静态流程为主；弱持久个人记忆与自动进化                                                      |


### 3.2 KyberKit 差异化五件事

> **上下文访问 × 工具执行 × 权限约束 × 节奏运行 × 能力沉淀**

其他竞品至少缺其中一条。「节奏运行」尤其重要：Cursor / Cowork 类产品本质是 IDE 内副驾，KyberKit 通过 Recurring / Triggered 合约把 Agent 拉到运营节奏中，成为按时间线值班的数字员工。

### 3.3 防御护城河

- **自进化技能**：从工作中生长出的技能 vs 纯安装型插件市场。  
- **组织数字资产**：CoW 技能流转 + 可追溯 fork 链。  
- **本土化通道**：企业微信（P1）→ 飞书（P2）等结构化 Gateway。  
- **信任与合规接口**：进化变更摘要、Checkpoint 回滚、权限与 OutputGuard（见第 7、8 节）。
- **任务级权限模型**：`UserPermission ∩ TaskPermission ∩ PolicyConstraint`，让企业 IT / 安全团队能理解、限制和审计 Agent 的行为边界。

## 4. Federation 架构模型（核心设计原则）

### 4.1 三层所有权

1. **Personal Identity**：KK.md、MEMORY.md、会话与轨迹 —— 默认不出域。
2. **Skill Capability**：Markdown + Frontmatter 技能；`private` → 经用户操作 `team` / `org`（3.5+）。
3. **Org Context**：外部编辑的知识库 MD、只读注入；不含个人对话原文。

### 4.2 技能「晋升」与 Copy-on-Write

- 团队使用的是 **Fork 后的副本**，演进互不覆盖（Memo L2）。  
- `forked_from`、`origin`、`author` 等字段在 Frontmatter 中预埋（详见第 8 节）。

### 4.3 隐私边界（产品承诺）

- KyberKit **不做**协作式知识库编辑；编辑在 Obsidian / 飞书等完成，仅同步内容注入。  
- 部署拓扑（Memo）：**纯中心化**；若未来支持私有化，在本节原则下扩展，不改变「身份层私有」契约。

### 4.4 Task Permission Contract

每个任务在执行前形成任务权限合约。它不是静态账号权限，而是对当前任务所需上下文、工具、风险等级、触发条件和审批要求的声明。3.0 起 KyberKit 支持三种合约形态：


| 合约形态                  | 触发模式                    | 典型场景                                 | 生命周期           |
| --------------------- | ----------------------- | ------------------------------------ | -------------- |
| **Ad-hoc** （即时合约）     | 用户主动发起一次任务              | "帮我诊断这个支付订单"                         | 单次执行后归档        |
| **Recurring** （周期合约）  | 按 cron / 自然语言时间         | "每天凌晨 1 点跑日报，连续 30 天"                | 长期合约，到期或被撤销才结束 |
| **Triggered** （触发式合约） | 事件流（告警 / 关键字 / Webhook） | "当 logs 出现 payment-fail-spike 时自动诊断" | 长期监听，触发后单次执行   |


三种合约共享同一份 schema 骨架，按形态加附字段：

```yaml
task_id: daily_biz_report.shawn
contract_type: recurring             # ad-hoc | recurring | triggered
actor: "shawn"
agent_role: "delegated_digital_employee"

# 仅 recurring 使用
schedule: "0 1 * * *"                # cron 或自然语言
expires_at: "2026-06-02T00:00:00+08:00"
scope_drift_limit:                   # 漂移阈值，超阈暂停并告警
  daily_token_budget: 50000
  failure_streak: 3

# 仅 triggered 使用
trigger:
  source: "logs.alert"
  match: "name=payment-fail-spike-* AND severity>=warning"
  backoff: "5m"                      # 节流避免风暴

# 三种合约共用
requested_context:
  - source: "payment.tx"
    scope: "my_visible_scope"
    mode: "read"
  - source: "feishu.bitable"
    scope: "项目看板/我管辖"
    mode: "read"
requested_tools:
  - tool: "insight.multidim.analyze"
    risk: "L0"
  - tool: "report.daily.generate"
    risk: "L0"
  - tool: "wecom.private_msg"
    risk: "L1"
  - tool: "wecom.send_card"
    risk: "L2"
effective_permission_rule: "user_permission ∩ task_permission ∩ org_policy"
approval_required:
  - "wecom.send_card"                # 即使在合约内, L2+ 仍单次审批
```

执行流程（三种合约统一）：

1. 用户提出任务（Ad-hoc 直接表达；Recurring / Triggered 由用户用自然语言描述节奏或触发条件）。
2. Agent 生成执行计划和合约草稿，包括 schedule / trigger / 漂移阈值。
3. Tool Permission Gate 计算有效权限。
4. 用户或策略批准需要确认的动作；长期合约一次授权产生持续生效的合约副本。
5. Agent 在合约范围内执行；扩权必须生成新合约或等待用户批准。
6. 每次执行（无论一次性或周期性触发）都将工具调用、权限判断、学习结果进入审计轨迹。

权限不足时，Agent 需要优雅降级：说明可访问哪些信息、缺少哪些权限、可生成哪些不完整产物，以及是否需要用户授权或邀请有权限的同事协作。

**长期合约的关键约束**（与 §4.5 联动）：

- **凭证绑定**：Recurring / Triggered 合约绑定用户 session token；token 过期或被撤销 → 合约自动暂停。
- **强制只读基线（3.0）**：3.0 阶段长期合约 **默认只允许只读 + L1 私聊推送**；任何 L2+ 写动作（群发 / 看板写入 / 客户回复）即使在合约范围内，也必须每次单独审批，不接受免审批白名单。
- **漂移检测**：Token 消耗 / 失败率 / 数据范围超阈值 → 合约自动暂停并通知用户。
- **撤销与回溯**：用户在 Web Console 一键撤销长期合约；撤销动作落审计，且不影响已执行结果。

### 4.5 防注入与扩权校验

`TaskPermission` 由 Agent 在计划阶段生成，存在被对抗 Prompt 提升权限边界的风险（例如知识库注入、工具回包伪装、用户消息伪造）。3.0 必须落地的最小防护：

- **TaskPerm 不得自我扩展**：合约一旦生成并经 Permission Gate 计算，执行阶段任何超出该集合的权限请求都视为新任务，必须重新进入合约流程并落审计。
- **不可变 deny-list**：内置一组默认禁止的高风险动作（例如管理员账户变更、批量外发、跨租户数据导出），不接受 Prompt 注入或工具回包改写；deny-list 升级走显式发布渠道。
- **高风险扩权需用户二次确认**：等级 ≥ L2 的工具调用，即使在合约范围内也必须在 IM / Console 中由用户点选确认；3.5 起可降级为团队批准。
- **审计不可绕过**：合约偏离、扩权请求、deny-list 命中均落入第 7.3 节的审计字段，作为信任与回溯的事实表。

防护接口在 3.0 即落地，以避免「合约存在但被绕过」的语义失效；策略与门槛的丰富度可由 3.5 / 4.0 增量演进。

### 5.1 3.0 — 个人节奏数字员工（2026 Q2 GA · 公司内 MVP）

3.0 把「单用户深度」升级为「单用户节奏」：Agent 不只是问一句答一句，而是按 OKR / 日报 / 告警 / 客服 / Spec / 周报 / 复盘的真实节奏陪跑。覆盖 §12 演示设计中 6 个 Act 的全部能力。

**P0 / P0.5（核心闭环）**：

- OutputGuard、Task / Permission Contract、Tool Permission Gate（含「保守 / 平衡 / 开发」默认策略包）、§4.5 防注入与扩权校验
- LearningLoop（按准入门槛放量）、Workspace Agent（后台异步）、Skill 渐进式披露 + 双来源技能（自省 + `/teach`）

**P1（节奏运行 + 上下文 + 形态）**：

- **三种合约调度器**：Ad-hoc / Recurring（cron + 漂移检测）/ Triggered（事件流监听 + 节流）
- **企业微信 Gateway（含审批卡片）**：结构化消息 + 审批按钮 + 进度回执 + 私聊推送，承载三种合约的审批与回调闭环
- **Web Console MVP**：任务台 + 权限合约视图（含周期合约监控面板）+ Evolution Changelog + 审计只读 + 技能库浏览 + 个人 OKR Graph（轻量版）
- **上下文 Connector 四源**：企业微信 / 飞书 / 产品文档库（含客服 SOP / 产研知识库）/ logs CLI
- **业务数据 Connector**：支付 MCP（贝易转）+ 用户行为日报（单独列出）
- **Capability Decomposition（LL 增强）**：用户用自然语言描述高层目标 → Agent 自动分解为 Skill 调用链 + Recurring/Triggered 合约草稿

**4.0 接口预埋**：

- `userId`、`SessionFactory`、存储 SPI、`GatewayMessage`、Skill 元数据（见 §8.7）
- `teamId` 占位 / 命名空间约定（为 4.0 Squad 预留）
- OKR Graph 边的 markdown frontmatter 协议（个人版即写入，4.0 升级为团队 Graph）

对外叙事：**一个按你节奏陪跑的私人数字员工 —— OKR、日报、告警、客户、Spec、周报全程在岗，所有动作可解释、可审计、可撤销。**

### 5.2 3.x — 个人版深化 + 4.0 接口预埋（2026 Q3 – 2027 Q1）

3.5 取消作为对外版本节点；中间发 3.0.x / 3.1 / 3.2 minor 承载以下增量：

- **dogfood 反馈迭代**：贝易转 dogfood 数据驱动的体验打磨（响应时延、合约漂移阈值、回滚体验）
- **Connector 拓展**：在 P1 四源之外，按业务节奏增加（如工单系统、邮件、会议系统）
- **Squad 接口预埋**：Per-User Agent Context、跨用户 @ 与转派的接口骨架；`teamId` 路径升级为非占位
- **OKR Graph 升级**：从「个人 Graph」逐步打开「团队 Graph」的只读视图
- **Spec 草稿增强**：从「暂存我的草稿箱」逐步打开「发给 owner 评审」入口（不接 CI）

> 3.x 不作为对外大版本节点的目的：让产品故事保持清晰二段式（个人 / 团队），减少销售与社区的概念负担。

### 5.3 4.0 — 团队中枢驱动（2027 Q1 GA）

把 3.0 的个人节奏扩展为团队节奏，让项目组的运营节奏由 Agent 编队驱动；保持每个员工的 Agent identity 与记忆私有（Federation 模型不变）。

**核心能力**：

- **Team Agent Squad**：多用户共享只读上下文池（OKR / 看板 / 知识库 / 业务数据），通过看板写入与 @ 转派协作；个人 KK.md / Memory 仍私有
- **OKR Graph（完整版）**：跨 OKR ↔ Project ↔ Task ↔ Metric 的多用户 Graph；自动维护与跨周变化追踪
- **站会决策辅助**：会议纪要 Connector + 决策 → 任务结构化 + 看板自动联动
- **跨人 TODO 转派 + 团队周报**：按分工自动加载每人 TODO；团队周报与 OKR 进度看板
- **Spec → 模块负责人审核流**：草稿 → 路由到 owner → 审核卡片 → 通过 / 驳回（**不接 CI**，CI 留 4.5+）
- **多租户 / RBAC / Audit Console**：ACL-aware Context Gateway、Org Policy Engine、Audit Console
- **技能市场（排序 / 推荐）**、协作知识库（L3）、L4 用户建模层（Memo Q5 延后项）
- `**ProviderRegistry`** 与模型特化 Prompt（正式多提供商）
- 飞书 Gateway 升级（覆盖卡片审批与会议纪要）、Telegram（可选）

对外叙事：**项目组的运营节奏由 Agent 编队驱动，每个员工保持自己的私人数字员工。**

### 5.3.x 4.5 / 5.0 — 产研自动化轴（2027 H2+）

4.0 之后单独划出「产研自动化轴」，作为 KyberKit 相对其他「企业 Agent 平台」的纵向跃迁，避开与 Cursor / Cowork 等 IDE-Agent 的横向竞争：

- **Spec → CI**：审核通过的 Spec 自动生成开发任务、触发分支创建、关联代码 Agent
- **代码 Agent 集成**：与 KyberKit Workspace Agent / 外部 IDE Agent 的协议对接（不必自研 IDE）
- **自动构建 → 开发 → 测试**：跨 Squad 的目标级联与依赖管理；Spec 评审 → Build → Test → Release 的全链路 Agent 协同
- **跨 Squad 级联**：从产研到客服到运营的目标 / 任务 / 数据 OKR Graph 级联

对外叙事：**KyberKit 把企业的产研运营节奏变成一张可治理、可审计、可演进的 Agent 网络。**

## 7. 信任机制：进化变更摘要（Evolution Changelog）

### 7.1 目标

企业用户采纳自进化前，必须能回答：**Agent 改了什么？依据是什么？能否撤销？**

### 7.2 能力要求

1. **可读变更摘要**：每次 LearningLoop 触发对 KK.md / MEMORY.md / Skill 等的修改，生成人类可读的摘要（diff 要点 + 自然语言说明）。
2. **可追溯**：关联 `trajectory.sqlite` / 轨迹记录，便于审计与调试。
3. **回滚**：利用现有 Checkpoint 能力，支持一键或按版本恢复。
4. **呈现层**：TUI / Gateway 均需能展示「最近进化」与详情（具体 UI 迭代分 sprint 实现）。

### 7.3 审计与责任边界

企业用户采纳数字员工前，还需要知道：**Agent 代表谁行动、依据什么权限行动、谁批准了高风险动作、产生了什么副作用。**

每次工具调用应至少记录：

- `actorUserId`
- `agentSessionId`
- `taskId`
- `toolName`
- `requestedPermission`
- `effectivePermission`
- `approvalStatus`
- `policyDecision`

这些字段是 4.0 Audit Console 的底层事实表，也是 3.0 可信执行体验的最小数据基础。

## 8. 3.0 技术决策汇总

以下内容与 [Kyberkit3.0-memo.md](./Kyberkit3.0-memo.md) 一致，本节为执行层速查；若冲突以 **本战略文档第 5–6 节（版本与优先级）** 为准。

### 8.1 Review 模型（Q2）

- 默认：`compactModel`（如 Haiku）。  
- Fallback：`reviewModel → compactModel → mainModel`。  
- 配置：`KyberConfig.model.reviewModel`。  
- 成本参考：100 活跃用户日均 Review 成本约从 $27/天 降至 ~$4/天（量级供规划用）。

### 8.2 多模型支持（Q3）

- 3.0 **正式支持 Anthropic**；其他提供商通过 ENV（OpenAI-Compatible baseURL）**非正式**切换，风险自负。  
- OutputGuard 预留 Provider 分支处理 tool_use 差异。  
- 4.0：`ProviderRegistry` + 模型特化 Prompt。

### 8.3 Gateway（Q4，优先级已上调企业微信）

- 顺序：**企业微信 → 飞书 → Telegram（可选）**。  
- `PlatformAdapter`：`text / card / file / action_response`；支持进度卡片与按钮。  
- **战略优先级**：企业微信 **P1**（见第 6 节）。

### 8.4 L4 用户建模（Q5）

- 3.0 **不纳入**；4.0 评估。理由见 Memo。

### 8.5 技能标准（Q6）

- 兼容 **agentskills.io** Frontmatter，并扩展 KyberKit 字段（`author`、`origin`、`forked_from`、`visibility`、`tags`、`usage_count`、`activation_paths`、`execution_mode` 等）。

| SessionFactory           | 工厂创建 AgentSession                 | Session Pool、并发                   | 工厂模式落地；测试覆盖 Session 创建、隔离与销毁                    |
| GatewayMessage           | 结构化消息类型                           | 飞书卡片等                             | 类型定义 + 至少 1 个 Adapter（企业微信）端到端验证                |
| **TeamSkillHub（接口预留）**   | 命名空间或目录约定、`teamId` 占位（待详细设计）      | **3.5** 团队技能协作                    | `teamId` / 命名空间约定写入 Frontmatter；占位接口与 stub 实现   |
| `taskId` / `actorUserId` | 每次任务与工具调用绑定用户和任务身份                | 审计、审批、权限计算                        | 任务 / 工具调用事件结构含两字段；审计日志可按字段查询                    |
| `permissionContract`     | 声明上下文、工具、风险等级、审批要求                | 任务级授权与策略治理                        | YAML schema 定义 + 计划阶段生成 + Permission Gate 消费链路  |
| `effectivePermission`    | 记录最终权限交集与策略裁决                     | 4.0 Policy Engine / Audit Console | 权限交集计算落表；审计日志含计算依据                              |


> 每条预埋点的 owner 与排期在 Sprint 计划中分配；上表的「完成标准」用作 3.0 GA 的 done-definition，避免 3.5 / 4.0 复用时发现接口不可用。

### 9.4 商业模型与 Unit Economics

3.0 的商业指标分两层：渠道健康与单位经济。商业模式已定为 **闭源商业许可 + 先公司内 MVP 验证**（见 §11.1），故 3.0 阶段不做对外定价；本节给出量级假设以支撑 3.5 PoC / 4.0 商业化时的定价决策。

**渠道健康（与 9.1 联动）**：

- **获客**：企业微信接入后的激活漏斗、Web Console 注册转化率
- **留存**：28 日留存（按渠道分段）
- **转付费**：试用 → 付费转化率（团队 / 企业档）

**Unit Economics 假设（100 用户 / 30 天 基线，量级供规划用）**：


| 项目                   | 假设值 / 区间               | 备注                           |
| -------------------- | ---------------------- | ---------------------------- |
| mainModel Token      | $0.6 ~ 1.2 / 用户/天      | 取决于任务密度与 Workspace 长任务比例     |
| reviewModel Token    | $0.04 / 用户/天           | 延续 §8.1 已测算结论                |
| Retrieval / FTS      | $0.05 / 用户/天           | 含本地索引计算与小规模向量库               |
| Tool I/O & Connector | $0.10 ~ 0.30 / 用户/天    | 企业微信 / 飞书 OpenAPI 调用、第三方限流摊销 |
| 运维 + 安全合规            | $0.15 / 用户/天           | 中心化部署摊销，含监控、审计、值守            |
| **合计成本**             | **~$1.0 ~ 2.0 / 用户/天** | 即 ~¥210 ~ 420 / 用户/月         |


**定价草案（3 档）**：


| 档位      | 目标客户               | 定价                   | 关键差异点                                            |
| ------- | ------------------ | -------------------- | ------------------------------------------------ |
| 个人 / 试用 | 单用户、效率先锋评估         | 免费 / 引流              | 单用户、私有 Skill、本地优先                                |
| 团队      | 2 ~ 10 人小组（PoC）    | ¥40 ~ 80 / 人 / 月     | Team Skill Hub、企业微信审批、技能 CoW                     |
| 企业      | 多部门 / IT / 安全 / 业务 | 席位 + 任务条数 + 审计模块（按需） | RBAC、Org Policy Engine、Audit Console、Connector 包 |


**毛利与回收期目标（草稿）**：

- **3.0**：不强求毛利转正；以验证 ICP / Unit Economics 模型为优先。
- **3.5（团队档）**：目标毛利率 ≥ 50%。
- **4.0（企业档）**：目标毛利率 ≥ 65%；企业 CAC 回收期 ≤ 12 个月。

**当前商业模式路径**（见 §11.1 决议）：闭源商业许可。这意味着 3.0 阶段不发布对外定价；3.5 PoC 起以「席位 + Connector 包」组合计费，4.0 引入「审计 / 治理模块」加价层。是否在某个版本节点开源核心 Runtime，按 §11.2 评估项决定。

## 10. 产品形态与增长策略

### 10.1 产品呈现形态

基于「企业数字员工的运行时与能力网络」定位，KyberKit 的产品形态不应停留在单一 TUI。TUI 保留为专业入口，但主体验应逐步演进为 **IM-first + Web Console + TUI** 的组合。


| 形态                       | 定位                          | 版本建议      |
| ------------------------ | --------------------------- | --------- |
| 企业 IM（企业微信优先，飞书第二）       | 主交互入口：任务发起、审批、结果通知、轻量追问     | 3.0 P1 起  |
| PC Web App / Web Console | 任务台、权限合约、审计、技能库、团队管理、管理员策略  | 3.5 起核心   |
| TUI / CLI                | 开发者与 power user 入口、调试、深度工作流 | 3.0 保留并增强 |
| PC 客户端                   | 本地文件、桌面自动化、跨应用动作            | 4.0 后评估   |


3.0 不重押 PC 客户端。客户端分发成本高、企业安全审查重；更合理的路径是：**企业微信做入口，Web Console 做控制台，TUI 做专业工作台**。

### 10.2 UI 布局与调性

KyberKit 应采用 **IM-first，但必须有独立 Web Console** 的界面策略。

企业微信 / 飞书适合：

- 发起任务
- 批准高风险动作
- 接收结果
- 对任务进行轻量追问

Web Console 适合：

- 查看任务历史与任务状态
- 对比执行计划
- 查看 Task Permission Contract（含 **Recurring / Triggered 长期合约监控面板**：cron 时间线、最近 7 次触发、漂移阈值、一键暂停 / 撤销）
- 审计工具调用
- 管理技能库
- 批量审批
- 配置团队工具白名单
- 回滚 Agent 学习结果
- **OKR Graph 面板**：3.0 个人 Graph（OKR ↔ Project ↔ Task ↔ Metric）；4.0 升级为团队 Graph，支持跨人 / 跨周期演进追踪
- **调度面板**：所有周期 / 触发任务的统一视图，含下次触发时间、最近运行结果、健康状态
- **合约草稿箱**：Capability Decomposition 生成的合约草稿、用户拒绝 / 修改的合约、长期合约到期前的续签提示
- **(4.0+) Spec 评审面板**：Spec 草稿 → 路由 → 模块 owner 审核 → 通过 / 驳回 → 自动建任务（CI 留 4.5+）

界面调性应为：**极简、可信、业务工具感**。更接近 Linear / Notion / Vercel / Slack Admin，而不是 Character.AI。企业用户购买的是信任、边界和交付能力，不是娱乐化人格。

### 10.3 Agent 拟人化边界

KyberKit 可以引入拟人特性，但必须克制。企业数字员工需要「有人味」，但不能像宠物或二次元助手。

拟人化分三层：


| 层级                         | 是否采用 | 说明                           |
| -------------------------- | ---- | ---------------------------- |
| 工作人格（Professional Persona） | 必须有  | 例如「谨慎」「高效」「先请示再执行」「默认生成草稿」   |
| 状态感 / 心跳                   | 可以有  | 展示 Agent 正在读取什么、等待什么审批、学会了什么 |
| 娱乐化聊天人格                    | 谨慎采用 | 不作为主卖点，避免降低企业场景的专业感          |


推荐的 Agent 状态表达：

- 「正在读取你有权限访问的 CRM 数据」
- 「等待你批准发送企业微信消息」
- 「本周我学会了 2 个新技能」
- 「这个任务缺少财务系统权限，已降级生成草稿」

产品原则：**Professional Persona，而非 Character Persona**。真正讨喜的数字员工不是会卖萌，而是可靠、透明、懂边界、能把事办完。

### 10.4 企业内运营推广策略

KyberKit 的企业增长不应从「平台」叙事开始，而应从高频具体任务切入，再沉淀为团队能力网络。

可参考的历史经验：


| 产品经验                    | 可借鉴点           | KyberKit 对应策略                          |
| ----------------------- | -------------- | -------------------------------------- |
| Cursor / GitHub Copilot | 从高频刚需切入，而非先讲平台 | 先打穿周报、数据分析、会议纪要、客户跟进等高频白领任务            |
| Slack / 飞书机器人           | 借现有工作流传播       | 通过企业微信结果分享、审批卡片、群内任务协作完成内部扩散           |
| Notion / Airtable       | 模板驱动增长         | 提供「销售周报数字员工」「财务报表分析数字员工」「招聘筛选数字员工」等模板  |
| ChatGPT                 | 个人先爽，再组织买单     | 3.0 让个人获得价值，同时提前具备权限合约与审计语言，降低企业 IT 阻力 |


数字员工模板不应只是 Prompt，而应包含：

- 任务类型
- 所需上下文
- 工具权限
- 输出格式
- 审批策略
- 可沉淀技能

### 10.5 版本化产品形态


| 版本       | 产品形态                                                                                       | 核心体验                                                   |
| -------- | ------------------------------------------------------------------------------------------ | ------------------------------------------------------ |
| **3.0**  | 企业微信审批入口 + Web Console MVP（任务 / 合约 / 调度 / 审计 / 进化 / OKR 轻量版）+ TUI 深入口                      | 个人节奏数字员工：按 OKR / 日报 / 告警 / 客服 / Spec / 周报陪跑，三种合约可解释可审计 |
| **3.x**  | dogfood 反馈打磨 + Squad 接口预埋 + OKR Graph 团队只读视图                                               | 个人版能力深化，4.0 接口逐步从占位升级为非占位                              |
| **4.0**  | 企业数字员工控制台 + Squad / OKR Graph / 站会 / 周报 / Spec 审核 + Context Gateway + Admin Policy Console | 团队中枢驱动：项目组运营节奏由 Agent 编队驱动，每个员工保留私人数字员工                |
| **4.5+** | Spec → CI + 代码 Agent 集成 + 跨 Squad 级联                                                       | 产研自动化：Spec → 开发 → 测试全链路 Agent 协同，跨 Squad 目标级联          |


阶段性爆点体验：

> 我在企业微信里派了一个任务，它知道自己能看什么、能做什么、需要我批准什么，最后把结果交付出来，还把这次做法沉淀成团队技能。

## 12. 3.0 GA 演示设计（贝易转 dogfood）

> 本章是 3.0 GA 的「最小可演示范围」与 dogfood 落点。所有能力都在 §6 P0 / P0.5 / P1 范围内，不引入额外承诺；Spec → CI、跨人 TODO 转派、团队 OKR Graph 等均明确划归 4.0 / 4.5+。

### 12.1 dogfood 业务上下文（贝易转）

[贝易转](https://beiyizhuan.example.cn) 是公司内的居住领域大额支付解决方案，覆盖：

1. **省心租 房租与服务费收款** —— 业主 / 租客的长周期收款，金额大、留痕严
2. **链家 / 贝联店东佣金收款** —— 跨城市、跨业态的商户结算，运营节奏明显
3. 公共能力：跨银行直连通道、城市 / 业态多维度运营、客服 / 风控 / 财务多角色协同

可介入的上下文（与 §11.1 #4 Connector 四源 + 业务数据 Connector 一一映射）：


| 上下文                        | 性质     | Connector 归属                      | dogfood 用途            |
| -------------------------- | ------ | --------------------------------- | --------------------- |
| 产品文档库（产品 / 系统设计、客服 SOP、QA） | 非结构化文本 | **产品文档库 RAG** (P1)                | 客服自动答复、设计问题快速回答       |
| 全量支付交易数据                   | 结构化    | **支付 MCP** (P1, 业务数据)             | 凌晨日报、告警诊断、城市 / 业态多维分析 |
| 系统日志（公司 logs 平台 CLI）       | 半结构化   | **logs CLI** (P1)                 | 异常诊断、告警跟进、版本回归对比      |
| 项目组工作推进看板（飞书多维表）           | 结构化    | **飞书 Connector** (P1)             | 站会准备、周报生成、TODO 个人视图   |
| OKR / 阶段目标                 | 结构化    | 飞书多维表 / 自维护 md (P1)               | 个人 OKR Graph、周报对齐、复盘  |
| 用户行为日报                     | 结构化    | **用户行为 Connector** (P1, 业务数据)     | 客户咨询热度、产品体验趋势         |
| 企业微信会话与文档                  | 非结构化文本 | **企业微信 Gateway / Connector** (P1) | 接收业务方咨询、推送审批 / 推送日报   |


### 12.2 演示主轴：「Shawn 的一周」（已决议 D1）

不分散讲单点能力，而是把 Agent 放进 PM 一周的真实节奏，6 个 Act 串成时间线。每个 Act 的 Skill 全部命中 §6 P1 清单。


| Act               | 时间        | 场景                | 触发模式      | 涉及 Skill / 合约                                                        | 真实可演 / 录制 |
| ----------------- | --------- | ----------------- | --------- | -------------------------------------------------------------------- | --------- |
| **A0 周一 09:00**   | 启动        | OKR 对齐与本周目标分解     | Ad-hoc    | OKR Graph（个人）+ Capability Decomposition                              | 真实可演      |
| **A1 周一 凌晨 1:00** | 自动        | 凌晨日报：交易 + 异常      | Recurring | 凌晨日报合约（cron 0 1 * * *）+ 多维分析 + 异常 detection                          | 录制 + 真实回放 |
| **A2 周二 09:30**   | 站会前 30min | 站会数据准备（多角度卡片）     | Ad-hoc    | 站会数据卡片 Skill + 多维交易 + 看板状态                                           | 真实可演      |
| **A3 周二 10:30**   | 站会后       | 决策 → TODO 落地（个人版） | Ad-hoc    | 会议纪要 → TODO（**个人范围**，跨人转派留 4.0）                                      | 真实可演      |
| **A4 周三 14:00**   | 触发        | 告警跟进 + 客户咨询答复     | Triggered | 告警跟进合约（logs.alert）+ 客服 SOP RAG + 草稿回复（**人审后发送**）                     | 真实可演      |
| **A5 周四 上午**      | 主动        | Spec 草稿（个人草稿箱）    | Ad-hoc    | Capability Decomposition → Spec 草稿（保存我的草稿，**不接 review / CI**，已决议 D2） | 真实可演      |
| **A6 周五 17:00**   | 周报        | 个人周报 + 进化总结       | Recurring | 个人周报 Skill + Evolution Changelog + 下周计划草稿                            | 真实可演      |


> 所有 Act 在企业微信 + Web Console 双入口都可看到合约、审计、回滚。LearningLoop 在 A2 / A4 / A6 真实地把方法沉淀为新 Skill / 偏好；不许用脚本伪造进化。

### 12.3 必备 Skill 清单（按 dogfood 切片）

把 GA 的 Skills 切成 3 个 Tier，按 sprint 优先级实现：

**Tier 0 · 节奏骨架（不可降级，否则 demo 不成立）**

- `**okr.daily.report`** —— 凌晨日报（Recurring，cron）
- `**stand_up.prepare**` —— 站会数据卡片（Ad-hoc）
- `**stand_up.decision_to_todo**` —— 决策结构化为个人 TODO
- `**alert.monitor.followup**` —— 告警跟进（Triggered）
- `**weekly.personal.report**` —— 个人周报 + 进化总结（Recurring）
- `**spec.personal.draft**` —— Spec 草稿写入个人草稿箱（D2）

**Tier 1 · 上下文与体验（dogfood-ready 必须）**

- `**okr.graph.maintain`** —— 个人 OKR Graph 维护（拓扑校验、跨周变化追踪）
- `**product.doc.qa**` —— 产品文档 RAG 答复（含客服 SOP）
- `**payment.tx.multidim.analyze**` —— 支付交易多维分析（城市 / 业态 / 通道 / 业务方向）
- `**logs.diagnose**` —— 日志切片诊断（按时间 / 关键字 / 服务）
- `**feishu.bitable.read**` —— 看板读取（项目状态 / TODO / 风险）
- `**wecom.draft_reply**` —— 客户咨询草稿（人审后发送）

**Tier 2 · 进化与自管理（public-ready 增强项）**

- `**learning.evolution.summary`** —— 进化变更摘要 / Changelog（§7）
- `**capability.decompose**` —— 自然语言目标 → Skill 链 + 合约草稿
- `**contract.health.monitor**` —— Recurring / Triggered 合约漂移检测
- `**memory.weekly.consolidate**` —— 个人记忆周度整理

### 12.5 公司内立项口径：贝易转「AI 降本」

KyberKit 在贝易转的 dogfood 同时是贝易转「**AI 降本**」战略的内部投入：

- **降低产品运营成本**：凌晨日报、告警跟进、客户咨询草稿、个人周报四类高频任务自动化，减少 PM / 客服 / 运营的重复劳动
- **提升决策质量**：OKR Graph + 多维交易分析让站会与周报的数据一致性显著提升
- **沉淀组织资产**：Skill / 偏好 / 工作流随 dogfood 自然沉淀，为 4.0 团队中枢驱动预备能力

> 该口径与 §0「公司内立项口径」、§11.1 #7 已决议项一致。

### 12.6 演示禁区与边界

为保证 GA 演示不被指控「卖飞机票」，明确以下边界：


| 不演示                     | 原因                           | 归属版本       |
| ----------------------- | ---------------------------- | ---------- |
| 跨人 TODO 转派 / 团队周报       | 个人版不做团队 Squad；4.0 才有完整模型     | 4.0        |
| Spec → 模块 owner 评审 / CI | 已决议 D2：3.0 仅写入个人草稿箱          | 4.0 / 4.5+ |
| Squad 共享上下文池            | 与个人 Identity Federation 边界冲突 | 4.0        |
| 自动写入业务数据库               | 3.0 业务数据 Connector 强制只读      | 长期         |
| 群发 / 客户最终回复 / 财务确认      | 始终走人审；§4.4「长期合约关键约束」         | 长期         |
| 自研代码 Agent / IDE 集成     | 与产研自动化轴绑定                    | 4.5+       |
| 脚本化 / 假数据自进化            | 信任根本，演示中的进化必须是真实日志           | 始终禁止       |


### 12.7 关键演示信号（成功标准）

GA demo 成功的最低门槛：

1. **节奏感**：观众在 6 个 Act 中至少 3 处自然产生「这就是我每周在做的事」共鸣
2. **可信感**：每次工具调用都能展示 Task Contract / 有效权限计算 / 审计入口
3. **进化感**：A2 / A4 / A6 中至少 1 处出现真实 Skill / 偏好沉淀，且能在 Evolution Changelog 中回看
4. **撤销感**：观众至少看到一次「一键撤销周期合约」或「回滚某次自进化」
5. **业务感**：贝易转真实业务数据（如某城市某通道近 7 日交易曲线、客服真实咨询）至少出现 1 次
6. **AI 降本感**：在结尾以 1 张图说明 dogfood 周期内（W-7 → W-1）相关任务的人时占用变化
