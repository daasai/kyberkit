# Kevin v1.5 产品需求文档 (PRD)

## 1. 产品愿景与定位 (Vision & Positioning)
**Kevin** 是 KyberKit 智能 Agent 能力网络（企业级多用户多场景协作集群）的**桌面终端与边缘节点**。
*   **愿景**：成为面向平台产研和职能用户的「数字化协同工作台」。辅助并接管用户日常的「标准工作」，彻底释放用户精力，使其能够专注于更具探索性和创造力的工作。
*   **定位**：不是一个传统的“套壳 AI 聊天窗口”，而是一个泛在的、具备极强上下文感知能力、隐去系统复杂度的**智能路由与执行网关**。

## 2. 核心设计哲学 (Design Philosophy)
*   **极简外观，无限延展 (The "iPhone" Paradigm)**：
    *   面向非程序员群体设计，提供极简、直觉式的交互界面。将背后极其复杂的跨系统调度、Agent 路由、大模型推理等过程完全包装、隐藏起来。
    *   在极简的交互之下，保留面向两端的极致拓展性（传感器 Sensors 与执行器 Actuators）。
*   **生命体特征：持续演进与自举 (Self-Evolution)**：
    *   Kevin 会随着用户的使用和成长而持续演进。
    *   用户可以从 KyberKit 网络中 Fork 现成的 Skills（技能），也可以自由创建。
    *   每个 Skill 在日常执行中通过反馈数据（LearningLoop）实现自我迭代与优化。

## 3. 核心能力模型：传感器与执行器的双向解耦 (Core Capabilities)
突破传统的“API 互联”思维，Kevin 将工作流抽象为人类数字孪生的“感官”与“四肢”：

### 3.1 传感器端 (Sensors - 泛在的感知与监听)
负责获取、接入和监听支撑用户工作上下文的所有触角。
*   **知识与文档感知**：读取项目/个人文档库（如飞书文档、本地知识库）的终端能力。
*   **系统状态感知**：读取与分析系统运行日志的 CLI。
*   **数据资产感知**：对接数仓查询接口、业务系统的指标看板。
*   **实时事件感知**：监听监控系统报警、企业社交媒体账号评论的 API。

### 3.2 执行器端 (Actuators - 意图的物理/数字投射)
负责能力输出、操作干预或是传递工作意图。
*   **内容生成与加工**：自动创建复杂的 Markdown 文档、CSV 数据表。
*   **可视化渲染**：直接生成并渲染用于汇报演示的动态 HTML 版 PPT。
*   **协同系统操作**：创建或推进 Kanban 任务进度（如 Jira、飞书项目）。
*   **业务系统干预**：执行后台系统（BOSS）的配置修改、下发某次营销活动策略。

## 4. 核心工作场景：数据到行动的闭环 (Standard Work Scenarios)
以**「单个产品（如：贝易转）的日常运营闭环」**为标准场景模版，Kevin 将接管以下全链路：
1.  **数据获取 (Observe)**：自动跨系统提取（数仓、飞书多维表格、企业微信沟通流等）。
2.  **加工与分析 (Orient)**：按照既定分析目标，对庞杂数据进行清洗、降噪与结构化。
3.  **洞察与决策 (Decide)**：在每日站会前，自动生成可供决策的洞察报告。
4.  **举措实施 (Act)**：
    *   向产研派发任务（优化功能/开发新功能）。
    *   自动调整营销计划。
    *   调整产品路由配置。
    *   更新/优化城市伙伴的 SOP 流程。
5.  **反馈监听 (Monitor)**：重新调用 Sensor 持续监控上述举措的数据反馈，形成闭环。

## 5. 边缘节点（本地）能力边界 (Edge / Local Capabilities)
作为 KyberKit 网络在用户桌面的“锚点”，本地设备（Kevin Client）承载以下独特使命：
1.  **本地设备执行能力集成 (The Local Execution Paradigm)**：
    *   代理用户身份，自动登录并穿透老旧的企业 Web 系统（类似 OpenClaw 机制）。
    *   提供本地沙箱环境，安全、高效地执行本地代码以处理大文件或涉密数据。
2.  **私有上下文的安全调用 (Privacy-Preserved Context)**：
    *   将高度敏感或私有的用户工作上下文保留在本地。
    *   在确保隐私与数据安全的前提下，完成本地信息与云端大模型的融合与脱敏交互。
3.  **端侧基础模型的温床 (Foundation for Edge AI)**：
    *   为未来本地化的小参数模型预留阵地。
    *   在端侧离线实现 OCR 识别、语音处理、甚至视频流的实时处理，降低中心网络延迟与保护绝对隐私。

## 6. 桌面级极简 GUI 交互隐喻 (The PC GUI Paradigm)
摒弃类似“飞书/钉钉”那种侧边栏堆砌、功能繁复的传统桌面软件范式。Kevin 在 PC 大屏上的交互必须像当年 iPhone 取消全键盘一样，实现**“界面复杂度的彻底收敛”**。收敛的方向不是“无界画布”（自由度过高，认知负荷太大），而是：

*   **轻量态（查询/快速任务）**：类 Glean 的极简唤起 + 结果卡片，用完即走。
*   **重度态（复杂工作流）**：类 Cursor 的严谨双栏工作台——中侧承载业务制品（Artifact），右侧展示 Agent 执行轨迹与对话。业务数据是主角，AI 退居配角。
*   **核心原则**：PC 大屏的优势用于展示直观结论和数据可视化，所有操作收敛为执行器（Actuator）的确认动作，流程细节对用户完全隐藏。对话、制品、长程任务在同一个心流中连续呈现，不强迫用户在 Tab 间跳转。

## 7. v1.5 界面设计与信息架构 (UI Architecture)

### 7.1 左侧导航栏（LeftSidebar）
左侧栏是空间导航与资产管理的枢纽，划分为四个逻辑区域：

**A. 顶部操作区**
| 元素 | 功能定义 | 展现形式 |
|---|---|---|
| **新建会话** | 发起新的智能对话流 | 核心 CTA 按钮 |
| **搜索** | **全局检索中心**：穿透检索已接入的 Sensor 数据、历史会话、历史 Artifact | 在 CenterPanel 展开沉浸式搜索结果 |
| **插件 (Skill Store)** | **能力集市**：浏览和下载 KyberKit Agent 网络分享的 Skills、Sensors 和 Actuators | 在 CenterPanel 展开商店视图 |
| **自动化 (Automation)**| **后台任务中心**：管理异步自动化任务（后台静默执行）；可调起本地无头浏览器处理私密任务 | 在 CenterPanel 展开任务仪表盘 |

**B. 文档库区（知识枢纽）**
这是 1.5 的重大架构升级，基于 KyberKit 的 Workspace 机制构建空间隔离的专属知识库。
*   **空间隔离**：每个 Space 拥有完全独立的文档库，不跨 Space 泄漏。
*   **格式兼容**：原生支持 Markdown, PDF, XLS, Word, PPT，打通旧有工作模式。
*   **双向联动**：
    *   *输入*：用户可自由浏览目录结构，通过 `@` 或附件选取任意文件作为对话上下文。
    *   *输出*：Agent 产出的新 Artifact 自动归档至该文档库，由 LLM 根据语义和用户习惯自动放置到合适目录，消灭"文件整理"动作。

**C. Sensor 状态区 & 历史区**
*   **已接入 Sensor**：展示当前 Space 挂载的外部感知器（如：数仓、监控系统），通过 🟢/🔴 状态灯指示健康度。
*   **历史会话**：按时间倒序排列当前 Space 的对话历史（未来迭代日期分组）。

**D. 底部（Space 切换器）**
*   **Vault 隐喻**：借鉴 Obsidian 设计，Space 切换器置于最底部（极小图标+名称）。
*   **多线程并发**：切换 Space 即**打开一个全新的独立应用窗口**。新窗口完整加载目标 Space 的文档库、Sensors 和对话流。支持多个 Space 窗口并行工作，上下文物理隔离。

### 7.2 顶部菜单栏（AppHeader）
**彻底移除 Tab 导航（如 Drafts/Published），回归纯粹的"操作条"，确保心流不中断。**

```text
[Kevin Logo]  [当前会话标题 - 可编辑]  ············  [🔔] [⚙️] [头像]
```

*   **当前会话标题**：展示/编辑当前任务名称。**未来彩蛋（动态灵动岛）**：预留为智能感知组件，后台任务执行时展示进度环，完成后短暂展开摘要，优雅收回。
*   **🔔 通知**：汇聚后台任务完成通知、系统告警、Sensor 离线提醒。
*   **⚙️ 设置**：唤起全局设置面板（配置 API Key、Sensor 授权等）。

### 7.3 中/右侧工作区 (Workspace Panels)
*   **右侧对话面板**：任务的"指令输入舱"与"过程追踪器"。
    *   **Input Toolbar**：包含 **📎 附件**（选文档库文件或传本地临时文件）和 **@ 提及**（精准召唤特定文件或 Sensor）。
    *   所有任务状态（后台流转进度、结果通知）均**内联**在对话流中展示，不强制跳转。
*   **中间制品面板**：任务的"结果展示台"。
    *   动态渲染生成的图表、文档或看板。
    *   配备专属的 **Artifact Toolbar**：包含 `Export (导出)`（PDF/MD/剪贴板）、`Share (分享)`（推至飞书/内部分享）、`编辑/预览`。

## 8. 开箱即用：初始化配置体系 (Onboarding)

### 8.1 开发者与用户两层配置架构
摒弃让普通用户手工编辑 `.env` 的高门槛做法。
*   **开发者层 (`.env`)**：由 IT 团队控制平台级策略，例如配置 `KYBER_MODEL_LIST` 设定公司允许接入的大模型清单。
*   **用户层 (GUI/AppData)**：用户仅需通过 UI 填入信息。配置加密持久化在系统级 AppData 中，热重载生效，绝不明文覆写 `.env`。

### 8.2 模型与网关接入策略
*   **统一协议**：底层统一切换至 OpenAI 兼容接口，废弃 `KYBER_MODEL_PROVIDER`。当用户填入自定义 Base URL 时，即被视为调用公司自建网关（兼容 VolcEngine 等）。
*   **内置预设**：默认自带 `DeepSeek V4 Flash`, `MiniMax M2.5`, `Qwen 3.x`, `GLM-5`, `Kimi K2.5` 等预设组合，切换模型自动补全对应 Base URL。

### 8.3 首次启动向导 (Onboarding Wizard)
强制全屏展示极简初始化卡片，阻断进入主界面，直至验证通过：
1.  **选择模型**：下拉框（由 `.env` 中的 `KYBER_MODEL_LIST` 动态渲染）。
2.  **输入 API Key**。
3.  *(可选展开)* **自定义接入地址**。
4.  **真实网络验证**：点击提交后，Sidecar 调用 `POST /config/validate` 发起真实 Ping 请求。仅当网络打通且 Key 有效时，方可进入主界面；失败则原位提示。

## 9. 三项核心功能定义 (Core Feature Harness)
为证明 Kevin 在专业场景下的"工业级可靠"，v1.5 必须将以下 3 个高频痛点场景打磨至极致。
> **"工业级可靠"标准**：Sensor 挂掉有明确降级通道；输出格式严格被模板契约约束；长任务有确定性进度反馈；制品全程可回溯。

### 9.1 场景一：站会数据准备 (Standup Data Brief)
**描述**：每日站会前，自动跨系统提取业务数据与工作看板，生成供会议决策的分析简报。
*   **依赖 Sensor**：飞书任务看板 (P0), 数仓 Query API (P1), 用户行为埋点 (P1), 企微咨询 (P1)。
*   **执行逻辑**：自动识别"昨日"时间窗，提取核心指标。
*   **降级策略**：若 P1 接口未就绪或超时，支持用户通过拖拽上传 CSV/Excel 文件进行"数据导入降级"。不因部分 Sensor 失败阻断整体流程。
*   **输出物**：严谨的 Markdown 数据表 + 关键异动高亮 + 3条洞察。

### 9.2 场景二：运营方案 / PRD 起草 (Decision to Document)
**描述**：基于会议决策结论，自动按照公司规范起草产品需求或运营方案草案。
*   **依赖 Sensor**：本地文档库 (P0, 获取公司模板), 场景一的产出 (P0, 自动注入上下文), 飞书历史文档 (P1)。
*   **执行逻辑**：Kevin 必须在文档库中命中预设模板（强模板契约）。若缺失模板，强制要求用户补充，拒绝大模型自由散漫发挥。
*   **输出物**：标准格式的 PRD/方案文档 Artifact，支持 CenterPanel 局部微调，支持一键 Share 推送至飞书文档。

### 9.3 场景三：周报生成 (Weekly Report)
**描述**：周末根据个人 OKR，汇总整周系统数据与任务进展，生成绩效周报。
*   **依赖 Sensor**：本地 OKR Markdown 文档 (P0), 飞书任务看板 (P0), 数仓/埋点周数据 (P1)。
*   **执行逻辑**：自动框定本周时间范围。严格读取 OKR 文档中的 KR 目标值进行差值计算。
*   **输出物**：实际 vs 目标对比表 + LLM 归因分析 + 下周重点计划。

### 9.4 共享基础设施优先级 (Infrastructure Priorities)
支撑上述场景的底座能力排期：
1.  **飞书 MCP Server (P0)**：三大场景全覆盖，无降级替代的核心。
2.  **本地文件 MCP 增强 (P0)**：支撑文档库树形结构、模板强匹配、多格式解析。
3.  **万能数据导入通道 (P0)**：所有 P1 外部系统接口的临时降级护城河。
4.  **内部业务系统 MCP (P1)**：数仓、埋点、企微接入。
