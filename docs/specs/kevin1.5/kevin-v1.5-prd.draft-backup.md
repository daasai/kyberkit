# Kevin v1.5 产品需求文档 (PRD) - 草案

## 1. 产品愿景与定位 (Vision & Positioning)
**Kevin** 是 KyberKit 智能 Agent 能力网络（企业级多用户多场景协作集群）的**桌面终端与边缘节点**。
*   **愿景**：成为面向平台产研和职能用户的「数字化协同工作台」。辅助并接管用户日常的「标准工作」，彻底释放用户精力，使其能够专注于更具探索性和创造力的工作。
*   **定位**：不是一个传统的"套壳 AI 聊天窗口"，而是一个泛在的、具备极强上下文感知能力、隐去系统复杂度的**智能路由与执行网关**。

## 2. 核心设计哲学 (Design Philosophy)
*   **极简外观，无限延展 (The "iPhone" Paradigm)**：
    *   面向非程序员群体设计，提供极简、直觉式的交互界面。将背后极其复杂的跨系统调度、Agent 路由、大模型推理等过程完全包装、隐藏起来。
    *   在极简的交互之下，保留面向两端的极致拓展性（传感器与执行器）。
*   **生命体特征：持续演进与自举 (Self-Evolution)**：
    *   Kevin 会随着用户的使用和成长而持续演进。
    *   用户可以从 KyberKit 网络中 Fork 现成的 Skills（技能），也可以自由创建。
    *   每个 Skill 在日常执行中通过反馈数据（LearningLoop）实现自我迭代与优化。

## 3. 核心能力模型：传感器与执行器的双向解耦 (Core Capabilities)
突破传统的"API 互联"思维，Kevin 将工作流抽象为人类数字孪生的"感官"与"四肢"：

### 3.1 传感器端 (Sensors - 泛在的感知与监听)
负责获取、接入和监听支撑用户工作上下文的所有触角。
*   **知识与文档感知**：读取项目/个人文档库（如飞书文档、Wiki）的终端能力。
*   **系统状态感知**：读取与分析系统运行日志的 CLI。
*   **数据资产感知**：对接数仓查询接口、业务系统的指标看板。
*   **实时事件感知**：监听监控系统报警、企业社交媒体账号评论的 API。

### 3.2 执行器端 (Actuators - 意图的物理/数字投射)
负责能力输出、操作干预或是传递工作意图。
*   **内容生成与加工**：自动创建复杂的 Markdown 文档、CSV 数据表。
*   **可视化渲染**：直接生成并渲染用于汇报演示的动态 HTML 版 PPT。
*   **协同系统操作**：创建或推进 Kanban 任务进度（如 Jira、飞书项目）。
*   **业务系统干预**：执行后台系统（BOSS）的配置修改、下发某次营销活动策略。

## 4. 核心工作场景：数据到行动的闭环 (Standard Work Scenarios)
以**「单个产品（如：贝易转）的日常运营闭环」**为标准场景模版，Kevin 将接管以下全链路：
1.  **数据获取 (Observe)**：自动跨系统提取（数仓、飞书多维表格、企业微信沟通流等）。
2.  **加工与分析 (Orient)**：按照既定分析目标，对庞杂数据进行清洗、降噪与结构化。
3.  **洞察与决策 (Decide)**：在每日站会前，自动生成可供决策的洞察报告。
4.  **举措实施 (Act)**：
    *   向产研派发任务（优化功能/开发新功能）。
    *   自动调整营销计划。
    *   调整产品路由配置。
    *   更新/优化城市伙伴的 SOP 流程。
5.  **反馈监听 (Monitor)**：重新调用 Sensor 持续监控上述举措的数据反馈，形成闭环。

## 5. 边缘节点（本地）能力边界 (Edge / Local Capabilities)
作为 KyberKit 网络在用户桌面的"锚点"，本地设备（Kevin Client）承载以下独特使命：
1.  **本地设备执行能力集成 (The Local Execution Paradigm)**：
    *   代理用户身份，自动登录并穿透老旧的企业 Web 系统（类似 OpenClaw 机制）。
    *   提供本地沙箱环境，安全、高效地执行本地代码以处理大文件或涉密数据。
2.  **私有上下文的安全调用 (Privacy-Preserved Context)**：
    *   将高度敏感或私有的用户工作上下文保留在本地。
    *   在确保隐私与数据安全的前提下，完成本地信息与云端大模型的融合与脱敏交互。
3.  **端侧基础模型的温床 (Foundation for Edge AI)**：
    *   为未来本地化的小参数模型预留阵地。
    *   在端侧离线实现 OCR 识别、语音处理、甚至视频流的实时处理，降低中心网络延迟与保护绝对隐私。

## 6. 桌面级极简 GUI 交互隐喻 (The PC GUI Paradigm)
摒弃类似"飞书/钉钉"那种侧边栏堆砌、功能繁复的传统桌面软件范式。Kevin 在 PC 大屏上的交互必须像当年 iPhone 取消全键盘一样，实现**"界面复杂度的彻底收敛"**。收敛的方向不是"无界画布"（自由度过高，企业场景认知负荷太大），而是：

*   **轻量态（查询/快速任务）**：类 Glean 的极简唤起 + 结果卡片，用完即走。
*   **重度态（复杂工作流）**：类 Cursor 的严谨双栏工作台——中侧承载业务制品（Artifact），右侧展示 Agent 执行轨迹与对话。业务数据是主角，AI 退居配角。
*   **核心原则**：PC 大屏的优势用于展示直观结论和数据可视化，所有操作收敛为执行器（Actuator）的确认动作，流程细节对用户完全隐藏。

## 7. v1.5 界面设计决策记录 (UI Design Decisions)

### 7.1 左侧导航栏（LeftSidebar）功能定义

**顶部操作区（四个导航按钮）：**

| 按钮 | 1.0 状态 | 1.5 定义 |
|---|---|---|
| **新建会话** | 已实现 | 保留，微调交互 |
| **搜索** | 空壳 | **全局搜索**：覆盖已接入的上下文（Sensors 内容）+ 历史会话 + 历史 Artifact 全文检索 |
| **插件** | 空壳 | **插件商店（Skill Store）**：浏览和下载来自 KyberKit Agent 网络分享的 Skills、Sensors 和 Actuators |
| **自动化** | 空壳 | **后台任务中心**：管理异步自动化任务（后台静默执行，完成后推送系统通知卡片）；本地可调起无头浏览器处理私密文件 |

**文档库区（原"上下文与资源"升级为"文档库"）：**

这是 1.5 的重大升级。底层 KyberKit 将在每个 Space/Project 的 Workspace 中设计专属的知识库目录。

*   **定位**：每个 Space 拥有独立的文档库，与 Space 隔离，不跨 Space 共享。
*   **文件格式支持**：全面兼容"PC 时代"和"互联网时代"的主流格式，包括 Markdown、PDF、XLS、Word、PPT 等，对接 Office/WPS 和飞书/企微的旧有工作模式。
*   **目录布局**：采用类 Obsidian 的文档目录树，用户可以自由浏览、展开目录层级。
*   **双向联动**：
    *   **输入方向**：用户可以从文档库中选取任意文件，作为当前 Chat 的上下文注入（@ 提及或附件选择）。
    *   **输出方向**：当 Agent 生成新的制品（Artifact）时，该制品自动出现在文档库目录中。存放位置由 LLM 根据现有目录分类结构和用户历史习惯自动决策，无需用户手动整理。

**Sensor 状态区（独立于文档库）：**
*   展示本 Space 已接入的 Sensor 列表，显示连接状态（在线 / 离线 / 错误）。
*   **设计待议**：Sensor 在 Sidebar 中的位置权重和可执行交互，待后续功能迭代中明确。

**历史区：**
*   历史会话按时间倒序排列，保留现有实现，后续考虑按日期分组。

**底部（Space 切换器）：**
*   参考 Obsidian 的 Vault 切换器范式，Space 切换器置于 Sidebar 最底部，以极小的图标+名称呈现。
*   点击后弹出 Space 列表（支持搜索、星标置顶）。
*   **切换 Space = 打开新的独立 App 窗口**，新窗口加载目标 Space 的全部上下文（文档库、Sensors、历史会话、Actuator 配置），两个 Space 窗口可并行运行，互不干扰。
*   原底部"Upgrade Plan"按钮移除；**设置入口（Settings）** 改为顶部右侧 ⚙️ 图标。

### 7.2 输入框工具栏（Input Toolbar）功能定义

| 图标 | 1.0 状态 | 1.5 定义 |
|---|---|---|
| **附件（attach_file）** | 空壳 | 支持两种来源：① 从"上下文与资源"中选择已授权的文件；② 临时上传本地文件或图片。上传内容直接作为**当前会话**的上下文（非全局注册）。 |


*   **移除**：`Drafts / Published / Reviews / Archive` 四个 Tab 彻底移除。
*   **移除**：`[Export]` 和 `[Share]` 从顶部 Bar 移出，下沉至中间面板制品工具栏（见 7.4）。

| 元素 | 定义 |

| **当前会话标题（可编辑）** | 显示当前 Chat 会话名称，点击可重命名。**未来彩蛋**：此区域预留为"灵动岛（Dynamic Island）"扩展位——LLM 可将其升级为一个智能感知的动态组件：后台任务执行中显示进度环，任务完成后短暂展开展示结果摘要，然后优雅收回，实现类 iPhone 灵动岛的桌面端体验。 |

### 7.4 中间面板制品工具栏（CenterPanel Artifact Toolbar）

*   包含操作：**导出（Export）**（Markdown / PDF / 复制到剪贴板）、**分享（Share）**（推送至飞书文档 / 分享给企业内 Kevin 用户）、**编辑/预览切换**。



## 8. 开箱即用：初始化配置设计 (Onboarding & Configuration)

### 8.1 设计原则
摒弃手动编辑 `.env` 文件的用户配置方式。采用**两层配置分离**：

*   **开发者/IT 层（`.env`）**：由公司技术团队部署时配置，控制平台级参数（允许使用的模型列表、SDK 协议等）。
*   **用户层（GUI）**：普通用户通过 Onboarding Wizard 或 ⚙️ 设置面板完成，仅需填写 API Key + 选择模型。配置以加密方式存储于 Tauri AppData 本地 JSON 文件，绝不明文写回 `.env`。

### 8.2 SDK 协议

**统一使用 Anthropic SDK**，不切换为 OpenAI 兼容客户端。

*   当用户填写了"自定义接入地址（Base URL）"时，理解为**"Anthropic SDK + 自定义企业网关"**，与当前 `.env` 中 `KYBER_MODEL_BASE_URL` 的用法一致（如 VolcEngine 代理）。
*   `KYBER_MODEL_PROVIDER` 字段**保留**，继续标识使用的 SDK 类型。

### 8.3 模型列表的两层控制

**开发者在 `.env` 中配置允许用户选择的模型列表**（新增字段 `KYBER_MODEL_LIST`）：

```env
# 开发者配置：用逗号分隔，定义用户侧下拉列表中可见的模型
KYBER_MODEL_LIST=deepseek-v4-flash,minimax-m2.5,qwen3.x,glm-5,kimi-k2.5

# 默认选中模型（用户首次打开 Onboarding 时的预选项）
KYBER_MODEL_DEFAULT=deepseek-v4-flash
```

用户看到的下拉列表由上述配置动态渲染，**不在前端代码中硬编码**。技术团队可按需调整公司可用模型范围，无需发版。

### 8.4 首次启动 Onboarding 流程

启动时 Sidecar 检测用户配置是否完整。若未配置，展示**全屏 Onboarding Wizard（不可跳过）**：

```
┌─────────────────────────────────────────────┐
│           👋 欢迎使用 Kevin                 │
│    选择你的 AI 模型，填入部门 API Key        │
│                                             │
│  模型   [ deepseek-v4-flash ▼ ]             │
│         （列表由技术团队在部署时配置）         │
│                                             │
│  API Key  [ ●●●●●●●●●●●●●  👁 ]            │
│                                             │
│  ＋ 自定义接入地址（公司网关用户展开）         │
│    [ https://...                     ]      │
│                                             │
│          [ 验证并开始使用  → ]               │
└─────────────────────────────────────────────┘
```

**验证机制**：调用 `POST /config/validate`，Sidecar 使用该配置发一次真实 ping：
*   成功 → 写入本地加密配置 → 进入主界面。
*   失败 → 原地显示可读错误（"API Key 无效" / "无法连接到该地址"），不跳出。

### 8.5 配置热重载（无需重启）

用户在 ⚙️ 设置中修改配置后，调用 `POST /config`，Sidecar 热重载 Anthropic Client，新配置立即生效。

### 8.6 新增 Sidecar 接口

| 接口 | 说明 |
|---|---|
| `GET /config` | 返回当前用户配置（Key 脱敏）+ 开发者配置的模型列表 |
| `POST /config` | 写入用户配置并热重载 Anthropic Client |
| `POST /config/validate` | 用给定配置发一次真实 ping，返回 `{valid: boolean, error?: string}` |

## 9. 三项核心功能定义 (Core Feature Harness)

> **"工业级可靠"的定义**：Sensor 挂掉有明确降级提示；输出格式严格遵循模板契约；长任务有进度反馈；所有生成的制品可回溯、可继续编辑。

---

### 9.1 Feature 1：站会数据准备 (Standup Data Brief)

**场景描述**：每天站会前，Kevin 自动跨系统拉取各项业务数据，汇总分析成可供决策的结构化数据报告，并结合飞书任务看板展示各工作项进展。

#### Sensor 接入策略

| Sensor | 数据内容 | 接入优先级 | 降级方案 |
|---|---|---|---|
| **飞书任务看板** | 各工作项状态、负责人、里程碑进展 | P0（必须真实接入） | 无降级，核心依赖 |
| **数仓 Query API** | 核心业务指标（GMV、转化率、DAU 等） | P1（尽力真实接入） | 用户手动导入 CSV/Excel 导出文件 |
| **用户行为埋点 API** | 漏斗分析、功能点击、路径数据 | P1（尽力真实接入） | 用户手动导入数据文件 |
| **企微咨询 API** | 用户反馈、客诉关键词、咨询量统计 | P1（尽力真实接入） | 用户手动导入聊天导出 |

#### Actuator / 输出

*   在 CenterPanel 渲染结构化"站会数据报告" Artifact，包含：
    *   各核心指标的昨日数据 vs 目标 vs 前日（表格 + 关键异动高亮）。
    *   飞书任务看板进展摘要（已完成 / 进行中 / 阻塞项）。
    *   LLM 自动生成"3 条关键洞察 + 1 条关注建议"。
*   报告格式严格遵循**预设模板契约**（固定结构，LLM 只填充数据，不自由发挥结构）。

#### 可靠性要求

*   **Sensor 超时/失败**：对应数据块显示"⚠️ 数据获取失败，可手动上传"，不因单个 Sensor 失败阻断整体报告生成。
*   **数据时间范围**：系统自动识别"昨日"时间范围，支持用户口语调整（如"帮我拉上周五的数据"）。

---

### 9.2 Feature 2：运营方案 / PRD 起草 (Decision to Document)

**场景描述**：站会决策后，用户录入分配的任务和目标，Kevin 基于决策背景和公司模板，起草对应的运营方案或产品 PRD。

#### Sensor 接入策略

| Sensor | 数据内容 | 接入优先级 |
|---|---|---|
| **文档库（本地）** | PRD 模板、运营方案模板（公司标准格式） | P0（已有本地文件 MCP，直接复用） |
| **Feature 1 输出** | 站会数据报告（自动作为背景上下文注入） | P0（会话内上下文传递） |
| **飞书文档** | 现有产品设计文档、历史 PRD（提供参考） | P1 |

#### Actuator / 输出

*   生成结构化 PRD 或运营方案 Artifact，严格遵循文档库中的对应模板。
*   支持用户在 CenterPanel 直接编辑后，一键推送至飞书文档（P1）。

#### 可靠性要求

*   **模板契约强制**：生成前必须先在文档库中找到对应模板；若未找到，提示用户先上传或创建模板，拒绝"自由发挥"生成。
*   **上下文衔接**：若当前会话包含 Feature 1 生成的数据报告，自动作为草稿背景注入，无需用户手动粘贴。

---

### 9.3 Feature 3：周报生成 (Weekly Report)

**场景描述**：每周末，Kevin 根据个人 OKR 文档，汇总本周各项业务数据，对比 KR 目标，整理工作进展，生成含洞察和下周计划的结构化周报。

#### Sensor 接入策略

| Sensor | 数据内容 | 接入优先级 |
|---|---|---|
| **个人 OKR（本地 Markdown）** | KR 目标值、时间周期（用户自己维护在文档库） | P0（本地文件 MCP，直接复用） |
| **飞书任务看板** | 本周已完成 / 进行中的工作项 | P0（与 Feature 1 共用飞书 MCP） |
| **数仓（周维度汇总）** | 同 Feature 1，时间粒度为一周 | P1（降级方案同 Feature 1） |
| **用户行为埋点（周维度）** | 同 Feature 1，时间粒度为一周 | P1（降级方案同 Feature 1） |

#### Actuator / 输出

*   在 CenterPanel 渲染结构化周报 Artifact，包含：
    *   **本周进展**：KR 实际值 vs 目标值对比表。
    *   **工作项盘点**：飞书看板已完成工作项列表。
    *   **洞察**：LLM 对本周完成情况的分析（超额 / 达标 / 落后及原因推断）。
    *   **下周计划**：基于本周落差和 OKR 剩余目标，生成下周重点工作建议。

#### 可靠性要求

*   **时间范围自动识别**：系统自动计算"本周一至今"，支持用户指定任意周范围。
*   **OKR 文档格式约定**：需在文档库中提供 OKR Markdown 模板，Kevin 按约定格式解析 KR 目标值；若格式不符，提示用户修正而不是静默出错。
