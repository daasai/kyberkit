# Kevin v1.5 UX 走查报告（基于当前界面截图）

- 日期：2026-05-06
- 走查角色：UX 负责人
- 评审依据：`docs/specs/kevin1.5/kevin-v1.5-prd-rev2.md`
- 评审对象：当前 Kevin 主界面截图（重度态）
- 评审方式：静态界面对照 PRD 条款走查（非交互录屏）

---

## 1. 结论概览

当前界面已具备 v1.5 三栏雏形（左导航、中制品区、右对话区），但与 PRD Rev2 仍存在多处关键偏差，主要集中在三类：

1. 核心信息架构偏差（Space 锚点缺失，底部被 Upgrade Plan 占位）
2. 任务状态可感知性不足（灵动岛四态不完整）
3. 产品心智风险（欢迎区内容暗示预装 Skill）

若不优先修正，将影响用户对 v1.5 的核心认知：Space 隔离、多窗口协同、异步任务与签批闭环。

---

## 2. 问题清单（按严重程度）

### P0 / High

#### 2.1 左下角 Space 锚点缺失，被 Upgrade Plan 占位

- 现状：左下为 `Upgrade Plan`，`Space / Vault` 位于左栏中上部。
- PRD 期望：Space 切换器作为左栏最底部固定锚点（Vault 隐喻）。
- 风险：用户从“工作空间导航”被引导到“商业升级入口”，破坏核心任务路径。
- 对照条款：§7.1E、§13。
- 建议：
  - 左下固定为 Space Switcher（当前 Space + 切换入口 + 新建 Space）。
  - 将 `Upgrade Plan` 移至设置或头像菜单，退出主任务动线。
  - 交互范式参考 Obsidian Vault Switcher。

#### 2.2 灵动岛未体现 v1.5 四态任务反馈

- 现状：顶部中间更像静态长条提示区，缺少进度环、待签批红色脉冲、完成瞬态摘要等状态语义。
- PRD 期望：空闲 / 执行中 / 待签批 / 完成瞬态 四态完整。
- 风险：异步任务与签批状态不可见，用户无法建立系统可控感。
- 对照条款：§7.2、§11.3、§10.2。
- 建议：
  - 空闲态显示可编辑会话标题。
  - 运行态显示任务名 + ETA + 进度环。
  - 待签批态显示红色脉冲与待处理数量。
  - 完成态展示 3 秒摘要后自动收起。

#### 2.3 欢迎区内容存在“预装 Skill”误导

- 现状：中栏欢迎区提供“站会数据 / 产品 Spec / RCA 分析”等建议入口。
- PRD 期望：v1.5 不预装任何 Skill，示例仅为契约蓝本。
- 风险：与产品范围冲突，造成预期管理失败。
- 对照条款：§9.0、§8.5。
- 建议：
  - 改为“如何开始”引导（@文件、附件、自然语言任务、Slash 调用已安装 Skill）。
  - 若展示样例，应明确“示例任务，不代表预装技能”。

### P1 / Medium

#### 2.4 Sensor 状态区信息不足，不满足最小可恢复闭环

- 现状：仅看到传感器列表，未见健康灯和恢复动作。
- PRD 期望：状态灯（绿/黄/红）+ 失败原因 + 最近成功时间 + 一键重连/重授权。
- 风险：故障不可诊断，可靠性心智下降。
- 对照条款：§7.1C、§15.1（Sensor 健康可见性 100%）。
- 建议：
  - 每个 Sensor 行增加状态灯和最近状态文案。
  - 点击展开卡片，展示失败原因与恢复按钮。

#### 2.5 历史会话区关键交互可发现性不足

- 现状：有标题+时间列表，但截图中未体现置顶/存档、6条折叠、待签批红点。
- PRD 期望：Codex 式行布局 + 悬浮操作 + 折叠策略 + 签批标记。
- 风险：历史管理效率低，签批提醒漏感知。
- 对照条款：§7.1D、§10.2。
- 建议：
  - 默认展示 6 条，补 `Show more`。
  - hover 明确显示置顶/存档图标。
  - 待签批会话行增加红色脉冲点。

#### 2.6 右侧面板结构偏离“对话与过程追踪器”定位

- 现状：右侧包含“对话/快速启动”标签与“制品”区域，结构语义混杂。
- PRD 期望：右侧以输入与过程追踪为主，中栏承载制品主视图。
- 风险：任务主副关系混乱，用户不清楚“在哪里看结果，在哪里下指令”。
- 对照条款：§7.3。
- 建议：
  - 右侧聚焦消息流 + 输入区 + 状态内联条目。
  - 制品主视图回归中栏，右侧仅保留制品跳转/引用。

#### 2.7 `Context & Sources` 区域信息架构不清，且未满足 PRD 的“文档库+Sensor”双目标

- 现状：`Context & Sources` 以单一扁平区块混合展示“本地文件入口”和“MCP/Sensor 项”，用户难以区分“可引用上下文”与“系统健康状态”。
- PRD 期望：左栏应显式支持文档库双向联动（可浏览、可 `@` 引用）以及 Sensor 状态区最小闭环（状态灯、失败原因、最近成功时间、重连/重授权）。
- 风险：
  - 用户在发起任务前无法快速建立“可用上下文”心智；
  - Sensor 异常时缺少可诊断和恢复入口，影响可靠性感知；
  - 区块体量与层级会挤压历史会话区，削弱会话管理效率。
- 对照条款：§7.1B、§7.1C、§7.3（Input Toolbar 的 `@` 提及语义）。
- 建议（最小可落地改造）：
  - 将 `Context & Sources` 拆分为两个子区：`文档库` 与 `Sensors`；
  - `文档库`：保留树形浏览，补“@引用到当前输入框”入口与最近引用；
  - `Sensors`：每项展示状态灯（🟢/🟡/🔴）+ 最近成功时间，点击展开失败原因与重连按钮；
  - 为两个子区提供独立折叠，优先保证“历史会话”可见面积。

### P2 / Low

#### 2.8 中右视觉权重失衡，主任务焦点不够清晰

- 现状：中栏留白较大，右栏容器与输入框存在感更高。
- PRD 期望：业务制品是主角，AI 交互是辅助手段。
- 风险：用户误判主工作区，降低复杂任务效率。
- 对照条款：§6.1、§7.3。
- 建议：
  - 增强中栏制品空态引导与默认结构骨架。
  - 降低右栏空态视觉噪声，突出“输入一次，结果在中栏”。

## 3. Space 区专项建议（对标 Obsidian）

建议将 Space 区按 Obsidian Vault Switcher 范式落地：

1. 左栏底部固定：`当前 Space 名称 + 下拉 + 新建`
2. 切换即开新窗口（符合 Space=独立窗口语义）
3. 列表项显示Space 名称
4. 当前 Space 高亮

## 4. 建议整改优先级（执行顺序）

### Sprint-1（必须）

- 左下 Space 锚点改造（替换 Upgrade Plan）
- 灵动岛四态完整化
- 欢迎区文案去“预装 Skill”误导

### Sprint-2（应完成）

- Sensor 状态区最小闭环（状态灯/失败原因/重连）
- 历史会话区补齐置顶/存档/折叠/签批红点
- `Context & Sources` 拆分为“文档库 + Sensors”并补齐 `@` 引用入口

### Sprint-3（优化）

- 右侧面板语义收敛
- 中右视觉重心与空态引导优化

---

## 5. 验收检查项（UI 侧）

- 左下固定 Space Switcher，且切换动作触发独立窗口
- 灵动岛在任务生命周期各状态下均有可见反馈
- 欢迎区不暗示任何预装 Skill
- Sensor 故障可被用户读懂并可执行恢复
- 左栏“文档库”与“Sensors”分区清晰，且支持 `@` 上下文引用
- 历史会话支持置顶/存档/折叠/签批标记
- 右侧作为对话与过程追踪器，中栏作为制品主舞台
