# Kevin (原 Kyberkit 3.0) MVP 产品规划与设计文档

> **文档状态**: 草案 (Draft)
> **更新日期**: 2026-05-03
> **产品代号**: Kevin (以区分之前的底层 Runtime 规划)
> **核心定位**: 贝壳内部 AI 工作日 —— 飞书生态之上的 AI 应用层与知识工作者生产力工作台。

## 1. 产品愿景与 MVP 定位

### 1.1 终极愿景 (North Star)
打造“企业上下文 × 工具链编排 × AI 生产力”的协作型工作台。它不是一个简单的对话框，而是深度融入员工日常工作流的“AI 驱动的对客指挥中心”（认知 + 行动 + 沉淀三个中枢），最终实现知识工作者的全面提效。

### 1.2 MVP (最小可行性/可被爱产品) 定位
**“麻雀虽小，五脏俱全”的终极形态微缩版。**
MVP 绝不仅是为了 CTO 汇报而造假的 Demo，而是基于真实底层架构（Tauri + Bun + MCP + Agent Runtime）、真实数据流、产生真实价值的可交付产品。

*   **目标用户**: 互联网平台部 PM、平台运营、架构师（首批 5-10 名真实种子用户）。
*   **核心目标**: 30 天内打赢 CTO 演示（拿到 Sponsor 通行证），同时让种子用户在日常工作中真实用起来，形成口碑（Plan C）。
*   **不做什么**: 不做全业态覆盖、不自建 IM/文档/BI/IDE、不碰一线经纪人场景、不自训大模型。

## 2. UI/UX 交互范式：三面板工作台 (左中右架构)

基于“非纯 Coding 场景”的生产力工具定位，Kevin 采用类似 Obsidian + Claudion 的左中右三面板架构，强调扩展性与产物中心：

### 2.2 面板功能定义

*   **左侧面板 (导航与上下文)**：
    *   承载文档/文件目录树（如本地项目文档库、飞书知识库映射）。
    *   历史对话列表与任务会话。
    *   **扩展性**: 支持未来添加不同的上下文源目录树。
*   **中间面板 (工作区 / Artifacts)**：
    *   **核心理念**: 工作的中心是“产物”，而不是“对话”。
    *   承载 Markdown 格式的 PRD 文档、数据报表、或 HTML 渲染的 PPT/卡片。
    *   支持富文本编辑（TipTap/CodeMirror），人机协同修改。
*   **右侧面板 (Agent 交互与状态)**：
    *   当前任务状态（Agent 思考轨迹、工具调用状态）。
    *   Chat Stream 窗口（与 Agent 的即时对话、指令输入）。
    *   **扩展性**: 支持不同 Tab，实现当前任务制品（如生成的中间数据）和会话的切换。

## 2.5 桌面端优先策略与技术选型

**强烈建议首期即引入桌面端形态。** 纯 Web 形态在浏览器沙箱限制下，无法真正成为“知识工作者的本地化工作台”。

### 2.5.1 为什么必须是桌面端？
1.  **本地文件系统级访问**: 架构师排查问题需要读取本地 Git 仓库、日志文件，PM 需要读取本地的 Markdown 模板。Web File API 体验极差且无法做到静默的 Agent 检索。
2.  **Bash 脚本与 CLI 调用**: 很多内部工具（如 `logs CLI`）或环境诊断需要直接执行 Shell 脚本。
3.  **系统级集成**: 未来可以实现全局快捷键唤起、读取剪贴板、甚至系统级通知。

### 2.5.2 最佳技术选型：Tauri 2.0 + React 19 + Bun Sidecar
*   **壳体 (Host)**: **Tauri 2.0**
    *   *优势*: 极度轻量（Mac 上打包体积仅几 MB，远小于 Electron），内存占用极低。基于 Rust，安全性高，且原生支持跨平台。
*   **前端 (Renderer)**: **React 19 + TypeScript + Vite + Tailwind v4 + shadcn/ui**
    *   *优势*: 现有的 `web/` 目录可以直接复用，生态极其丰富，开发效率最高。
*   **本地服务 (Sidecar)**: **Bun (编译为单二进制文件)**
    *   *优势*: 现有的 `src/console-server/` 已经是基于 Bun 开发的 HTTP/SSE 服务。Tauri 可以将 Bun 编译后的单文件作为 Sidecar 随应用一起打包分发。前端通过 localhost 端口与 Bun Sidecar 通信，Bun 负责运行 AgentLoop、读写 SQLite 轨迹、调用本地 Bash 和 MCP Server。
    *   *架构隔离*: 这种架构让 Tauri 只负责“壳”和“系统 API”，Bun 负责“AI 逻辑和数据”，前端负责“渲染”，职责极其清晰。

## 3. 核心演示场景：贝易转 Dogfood (MVP 范围)

MVP 采取“一刀致命”策略，以**贝易转产品**的真实工作流为演示主轴（参考《Shawn 的一周》），证明 AI 降本与决策提效。

### 3.1 演示上下文 (MCP 数据源)
*   **飞书文档库**: 产品/系统设计、客服 SOP。
*   **内部数据 BI**: 交易数据、用户行为数据（通过只读 MCP 接入）。
*   **系统监控与日志**: 异常诊断、告警跟进。

### 3.2 核心演示切片 (Act 裁剪)
*   **场景 A：站会与数据准备 (面向 PM/运营)**
    *   Agent 自动读取飞书多维表看板、调用 BI 数据 MCP 获取昨日交易/行为数据。
    *   在中间面板生成结构化的“站会数据卡片”和“凌晨日报”。
*   **场景 B：告警跟进与 RCA (面向 架构师/技术 PM)**
    *   输入报错 TraceID，Agent 检索系统日志与代码库。
    *   在中间面板起草 RCA 报告，右侧展示分析轨迹。
*   **场景 C：PRD/SOP 智能工坊**
    *   基于飞书文档库的 RAG，快速生成客服 SOP 回复草稿，或基于会议纪要生成 PRD。

## 4. 种子用户冷启动 (Onboarding)

为了让首批 5-10 名种子用户迅速体验到 "Aha Moment"，降低配置门槛：

*   **预置开箱即用模板**: 
    *   在本地（或通过内置的 Local File MCP）预先打包好几个高频的企业模板（如《贝壳 PRD 标准模板》、《RCA 报告模板》、《站会数据卡片模板》）。
    *   用户无需手动配置复杂的 MCP 权限，只需点击“一键运行”对应的 SOP 快捷指令，Agent 即可基于预置模板和基础输入生成产物。
*   **渐进式授权**: 
    *   初期仅需用户授权最基础的飞书只读权限或本地目录读取权限，即可跑通核心模板。

## 5. Kevin MVP 首期功能地图 (Functional Map)

为了在 30 天内交付具有“一刀致命”演示效果且能让种子用户真实使用的工作台，我们将功能严格裁剪并划分为四大模块：

### 5.1 核心工作区 (Center Panel: Artifacts & Editor)
这是“工作台”区别于“聊天框”的核心，承载最终产物。
*   **[P0] Markdown 富文本渲染器**: 支持 GFM (GitHub Flavored Markdown)、表格、代码块高亮 (shiki)，用于渲染 PRD 和 RCA 报告。
*   **[P0] 基础人机协同编辑**: 允许用户在生成的 Markdown 产物上直接手动修改文本。
*   **[P1] 行内 AI 编辑 (Cmd+K)**: 选中某段文本，唤起浮窗让 Agent 进行“扩写、缩写、语气调整、逻辑补充”。
*   **[P0] 产物导出**: 支持一键复制为 Markdown 或纯文本，方便用户粘贴回飞书文档。
*   **[P1] 多 Tab 切换**: 支持同时打开多个产物（如同时看 PRD 和数据卡片）。

### 5.2 智能助手与交互 (Right Panel: Agent & Chat)
承载用户的意图输入与 Agent 的思考轨迹展示。
*   **[P0] 基础对话流 (Chat Stream)**: 支持多轮对话，流式输出回复。
*   **[P0] 上下文显式引用 (@ 提及)**: 在输入框输入 `@`，可快速引用左侧的飞书文档、本地文件或模板作为当前对话的强制上下文。
*   **[P0] Agent 轨迹可视化 (Trajectory)**: 真实展示 Agent 正在执行的动作（如：`[正在调用 Feishu MCP 读取文档...]` -> `[正在思考...]` -> `[生成中...]`），建立信任感。
*   **[P1] 任务中断与重试**: 允许用户随时打断 Agent 的生成过程，或修改 Prompt 后重试。

### 5.3 上下文与资源管理 (Left Panel: Context & Navigation)
管理企业知识库与历史资产。
*   **[P0] 历史 Artifacts 列表**: 按时间倒序排列最近生成的产物会话（如“Q2 支付系统 PRD”、“5月3日站会数据”），点击可恢复工作区。
*   **[P0] 预置模板库 (Templates)**: 内置写死的 3-5 个高频模板（PRD 模板、RCA 模板、客服 SOP 模板），点击即可快速创建对应的 Agent 任务。
*   **[P1] 资源目录树 (Resource Tree)**: 可视化展示当前连接的 MCP 资源（如：已授权的飞书知识库目录结构、已绑定的本地 Git 仓库路径）。

### 5.4 底层支撑与 MCP 接入 (Infrastructure)
不可见的底层能力，但决定了产品的真实性。
*   **[P0] 本地轨迹存储 (SQLite)**: 所有的对话、生成的产物、Agent 思考轨迹均保存在本地 `.kyberkit/` 目录下，保障数据隐私与审计需求。
*   **[P0] Local File MCP (内置)**: 允许 Agent 安全地读取用户指定的本地 Markdown 模板或代码文件。
*   **[P0] Feishu Doc MCP (MVP 版)**: 接入飞书开放平台，允许 Agent 根据 URL 读取特定的飞书文档内容（仅需只读权限）。
*   **[P1] 简易 Git MCP (或 CLI 调用)**: 允许 Agent 执行基础的 `git log` 或 `grep` 命令，辅助 RCA 场景的代码检索。

## 6. 30 天关键里程碑

*   **Week 1**: 确定左中右三面板 UI 架构，完成 Tauri + React 桌面壳的基础搭建，跑通本地 Bun Sidecar 通信。
*   **Week 2**: 实现 1-2 个核心的只读 MCP Server (如 Local File 模板读取, 简易飞书文档读取)。
*   **Week 3**: 串联贝易转场景的 Agent Workflow，实现端到端的数据流转，在中间面板成功渲染 Markdown/数据卡片。
*   **Week 4**: 种子用户试用 (5-10人)，基于预置模板体验，收集反馈，打磨 Demo 剧本，准备 CTO 汇报材料。
