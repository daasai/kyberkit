# Kevin (Kyberkit 3.0) MVP: Sprint Implementation Plan (v2)

根据您的反馈，我们**彻底废弃旧有的 `console-server` 和 `web` 目录**，将其作为历史验证产物。本次 MVP 将是一次从底层向上重新构建桌面端架构的“破局之战”。我们只从旧代码中“按需提取”纯粹的 Agent Runtime 核心引擎代码（如 LLM Client、基础 Prompt 模板）。

本次计划继续聚焦于端到端打通 **Scenario A (获取站会数据 -> 生成产品升级 Spec)**。

---

## 1. 全新架构演进评估报告 (重构版)

由于不再被旧包袱掣肘，这赋予了我们极高的架构自由度。当前的核心挑战由“系统迁移”转变为**“桌面端高内聚架构的从零构建”**：

### 1.1 纯净的 Tauri + Bun Sidecar 架构 (高风险 & 高收益)
*   **重构方案**: 
    *   直接使用 Tauri 2.0 CLI 初始化全新的前端工程（React 19 + Vite + Tailwind v4）。
    *   在全新目录下（如 `src-sidecar`）从零编写一个极简、专注于和 Tauri 通信的 Bun HTTP/SSE 服务。
    *   **架构优势**: 彻底摆脱原 `console-server` 的历史负担，实现前端、Tauri 壳、AI 后端的三者清晰解耦。
*   **挑战**: 需要从头处理 Sidecar 的自动启动、端口动态分配和进程终止逻辑，防止僵尸进程。

### 1.2 UI 范式重写：Artifact-centric (中风险)
*   **重构方案**: 彻底抛弃旧版 `web` 的代码，基于 shadcn/ui 等现代组件库，从第一行代码开始按照“左（导航）、中（产物）、右（对话）”的三面板理念进行组件树划分。
*   **挑战**: 产物区（Center Panel）需要高定制化的 Markdown 渲染组件，支持代码高亮与表格，并且需要预留未来的行内编辑（Cmd+K）接口。

### 1.3 数据流与持久化 (中风险)
*   **重构方案**: 废弃可能存在于 `console-server` 中的复杂中间件，在新的 Sidecar 中直接建立轻量级的本地 SQLite 连接，仅存储基础的会话与产物记录。通过全新编写的 `Local File MCP` 使得引擎直接读取本地文件。

## 2. Kevin MVP 4 周 Sprint 规划 (重构演进版)

### Week 1: 纯净底座与架构连通 (The Clean Base)
**目标**: 从零搭建起 Tauri + React 19 + Bun Sidecar 的骨架，跑通最简单的对话流通信。
*   **任务 1.1**: 执行 Tauri 初始化流程，搭建基于 Vite + React 19 + Tailwind v4 的全新前端目录结构。
*   **任务 1.2**: 创建极简的 Bun Sidecar (HTTP/SSE)，编写 Tauri 启动/退出时管理该进程的 Rust/Node 粘合代码。
*   **任务 1.3**: 在前端实现基础的三面板静态排版（左侧目录区、中间占位区、右侧对话输入区）。
*   **任务 1.4**: 引入核心 Agent Runtime 库，打通从前端发请求 -> Sidecar 执行最简单 LLM 调用 -> SSE 流式返回到右侧面板的闭环。

### Week 2: UI 组装与本地上下文打通 (Context & Layout)
**目标**: 界面具备交互能力，AI 能够读取本地模板。
*   **任务 2.1**: 开发 `Local File MCP` 工具，允许 Agent 安全读取本地 `.md` 模板文件。
*   **任务 2.2**: 引入 `react-markdown` 等依赖，在中间面板实现高质量的 GFM 和表格渲染能力。
*   **任务 2.3**: 在左侧面板实现硬编码的“预置模板库”（如：产品升级 Spec 模板）和资源目录视图。
*   **任务 2.4**: 实现右侧 Agent 聊天面板的 Trajectory（思考轨迹）UI 原型，使其能显示正在调用工具的状态。

### Week 3: 核心场景攻坚 (Scenario A)
**目标**: 针对站会数据到产品升级 Spec 的端到端业务流。
*   **任务 3.1**: 开发 `BI Data MCP`（MVP 阶段对接真实测试库或模拟接口），使 Agent 能获取昨日业务数据。
*   **任务 3.2**: 编排 Agent Workflow：接收“生成昨日站会数据卡片”指令 -> 调取数据 -> 在中间面板渲染表格卡片。
*   **任务 3.3**: 实现场景延续衔接：指令“基于这份数据，生成产品升级 Spec” -> Agent 读取本地 Spec 模板 -> 结合上一步的数据 -> 在中间面板覆写并渲染完整的 Markdown Spec。
*   **任务 3.4**: 实现产物在中间面板平滑加载渲染（流式更新效果）及右侧状态同步。

### Week 4: 封版与体验打磨 (Polish & Build)
**目标**: 具备给种子用户分发和 CTO 演示的完成度。
*   **任务 4.1**: 接入本地 SQLite，实现会话历史和生成产物的本地持久化存储与左侧列表加载。
*   **任务 4.2**: 实现快速启动指令区（Onboarding），用户一键点击触发 Scenario A 完整流程，无需手动打字。
*   **任务 4.3**: Tauri 生产环境打包验证：在 Mac 环境下生成单机的 `.dmg` 或 `.app` 包，确保 Sidecar 可执行文件被正确绑定且无环境依赖运行。
*   **任务 4.4**: 修复高优交互 Bug，准备 CTO 演示剧本。

## 3. User Review Required (需要您确认的决策)

> [!IMPORTANT]
> **关于 Tauri UI 开发的隔离策略**
> 由于我们是彻底重写，为了在 Week 1 保持最高开发效率，我建议在前期先将前端（React）和后端（Bun Server）作为**纯 Web 模式独立运行开发**（`npm run dev`），暂不涉及 Tauri 壳的复杂调试。直到周五，再将前端 build 后塞入 Tauri 进行连通性测试。这能极大避免开发初期在 Rust 和系统级环境问题上陷入泥潭。请确认是否同意这种节奏？

> [!NOTE]
> **旧代码的废弃处理**
> 对于旧的 `console-server` 和 `web` 目录，我们是直接执行删除操作（`[DELETE]`），还是保留它们仅作为参考代码，后续单独建一个全新的项目仓库？建议在这个项目中直接重命名为 `console-server.bak` 和 `web.bak` 以免干扰我们的新建工程。
