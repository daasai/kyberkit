# 第 6 章：Phase 3 —— 智能任务编排

当 Agent 需要处理复杂、多步骤的任务时，单纯的线性对话（CoT）往往难以为继。KyberKit 的 **Phase 3 (Intelligence)** 引入了**确定性工作流引擎**，将模型生成的模糊计划转化为严谨的任务图 (TaskGraph)。

## 6.1 从线性到图：TaskGraph (DAG)

在 KyberKit 中，复杂的任务被建模为 **有向无环图 (Directed Acyclic Graph, DAG)**。
- **节点 (Node)**：单个原子任务（如“读取文件”、“运行测试”）。
- **边 (Edge)**：任务之间的依赖关系（如“必须先运行测试，才能提交代码”）。

### 为什么需要 DAG？
传统的 Agent 如果陷入逻辑死循环，会不断消耗您的 API 额度而不产生结果。KK 的工作流引擎在执行前会对图进行**拓扑排序 (Topological Sort)**，并在物理层面拦截任何逻辑循环。

## 6.2 确定性执行引擎与 Kahn 算法

KyberKit 内部集成了基于 **Kahn 算法** 的执行引擎。
1.  **静态分析**：在执行前检查 TaskGraph 是否存在环路。
2.  **分层并行**：识别哪些任务是互不依赖的，从而在支持的运行时（如 Bun）中实现无锁并行执行。
3.  **确定性回卷**：如果某个节点失败，引擎会精准定位依赖链并停止受影响的分支，而不会盲目重试整个图。

## 6.3 上下文预算管理 (Context Budgeting)

模型的能力受限于上下文窗口（Context Window）。KyberKit 的 **ContextBudget** 模块负责智能地管理这些宝贵的“空间”。

### 声明式上下文绑定
您可以为不同的任务定义不同的上下文优先级：
- **P0 (强制)**：当前任务目标、AGENTS.md 中的核心准则。
- **P1 (重要)**：最近的 3 次轨迹记录、相关的代码片段。
- **P2 (可选)**：项目背景文档、较远的历史会话。

当窗口即将溢出时，KK 会基于优先级自动剔除 P2 和 P1 的内容，确保 **P0 (关键指令)** 永远不会被截断，从而避免模型产生由于上下文残缺导致的幻觉。

## 6.4 任务规划器 (Planner) 的角色

在 Phase 3 中，Planner 的职责是：
1.  **需求解析**：将用户的模糊指令转化为 `TaskGraph` 结构。
2.  **动态重计划 (Re-planning)**：当环境反馈（如工具报错）与预期不符时，生成补丁图（Patch Graph）并合并到主任务图中。
